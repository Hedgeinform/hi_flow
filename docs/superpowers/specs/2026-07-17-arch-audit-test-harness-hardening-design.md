# Дизайн: исправление долгов тестовой обвязки ArchAudit

**Дата:** 2026-07-17
**Статус:** утверждён оператором, ожидает реализации
**Контур:** `hi_flow/skills/arch-audit/`

## Контекст

После исправления JavaScript/ESM-регрессии полный прогон выявил четыре старых дефекта тестовой обвязки:

1. часть тестов разрешает `tests/fixtures/...` через текущий рабочий каталог;
2. четыре интеграционных теста пишут отчёты в версионируемые каталоги фикстур;
3. D9-парсер и Markdown-фикстуры чувствительны к CRLF в Windows worktree;
4. AJV игнорирует `format: date-time`, поэтому `audit_timestamp` фактически не валидируется.

Первый и второй дефекты нарушают изоляцию тестов. Третий делает результат зависимым от платформы. Четвёртый ослабляет уже объявленный D8-контракт. Нового пользовательского поведения эта работа не вводит.

## Принятое решение

Исправить причины в каждом слое, не использовать глобальную смену `process.cwd()` и не создавать новый универсальный тестовый фреймворк.

### Независимые пути к фикстурам

Создать маленький тестовый helper, который вычисляет корень пакета и пути к фикстурам относительно собственного `import.meta.url`. Тесты, которые сейчас передают строковые пути `tests/fixtures/...` или строят их от `process.cwd()`, переводятся на helper.

`vitest.config.ts` получает абсолютный `root`, вычисленный относительно собственного `import.meta.url`, чтобы discovery тестов и конфигурация также не зависели от места запуска. Значение `root: '.'` запрещено как сохраняющее зависимость от `cwd`. Боевой код не импортирует тестовый helper.

### Изолированные интеграционные артефакты

Интеграционные тесты используют исходные проекты в `tests/fixtures/<project>` только для чтения. Общий тестовый helper создаёт отдельный выходной каталог через `mkdtemp()` в системной временной директории, передаёт точный созданный путь callback-функции и удаляет только этот путь в `finally`.

Cleanup выполняется и при успешном callback, и при любом исключении внутри него, включая упавший assertion. Ошибки `rm()` не перехватываются и не проглатываются. Отдельный тест helper'а вызывает исключение внутри callback, проверяет повторный выброс исходной ошибки и отсутствие созданного каталога после завершения.

Точный integration-контур:

- `tests/integration/barrel-project.test.ts` — убрать запись в tracked `audit-report/`;
- `tests/integration/cycle-project.test.ts` — убрать запись в tracked `audit-report/`;
- `tests/integration/god-object-project.test.ts` — убрать запись в tracked `audit-report/`;
- `tests/integration/layered-project.test.ts` — убрать запись в tracked `audit-report/`;
- `tests/integration/js-esm-project.test.ts` — сохранить временный `outDir`, но перевести cleanup на тот же гарантированный helper.

RED не запускает старые мутирующие тесты прямо на рабочем дереве. Вместо этого structural regression-тест читает исходники четырёх проблемных тестов и сначала падает на tracked `outDir`. После GREEN внешний gate снимает SHA-256 baseline списка и содержимого всех tracked `tests/fixtures/**/audit-report/**` до полного suite, повторяет снимок после и требует полного равенства. Это корректно работает и в исходно грязном worktree, поскольку сравнивает состояние до/после, а не требует общего `git status --porcelain == empty`.

Обновление эталонных снимков, если оно когда-либо потребуется, остаётся отдельной явной операцией и не выполняется обычным тестовым прогоном.

### Переводы строк D9

`loadD9()` нормализует входной Markdown `\r\n` и одиночные `\r` в `\n` сразу после чтения. Это устраняет платформенную зависимость на границе парсера.

В корневой `.gitattributes` добавляется точное правило `hi_flow/skills/arch-audit/tests/fixtures/**/*.md text eol=lf`. Нормализация в коде остаётся необходимой защитой для внешних D9-документов, которые могут приходить с CRLF независимо от Git.

Gate запускает `git check-attr text eol -- <каждая Markdown-фикстура>` и требует `text: set`, `eol: lf`. Отдельная байтовая проверка индексированных Markdown-фикстур требует отсутствия `0D` перед `0A`; runtime-тест не может заменить этот Git-level gate.

### Полная D8 date-time валидация

Добавить runtime-зависимость `ajv-formats` и зарегистрировать её на существующем экземпляре AJV до компиляции D8-схемы. Сама схема и её версия не меняются: `format: date-time` уже является частью контракта.

Невалидная строка `audit_timestamp: 'not-a-date'`, которая нарушает только формат, должна отклоняться. Форматированная ошибка валидации сохраняет `keyword` AJV; тест требует `keyword: 'format'`, путь `/metadata/audit_timestamp` и сообщение о `date-time`. Валидные RFC 3339 значения, включая результат `new Date().toISOString()`, продолжают проходить.

Финальный тестовый вывод не должен содержать прежний AJV warning `unknown format "date-time" ignored`. Это проверяется по реальному stdout/stderr полного запуска, а не только положительным unit-тестом.

## Проверки

Реализация выполняется через четыре независимых RED → GREEN цикла:

1. внешний запуск сначала не находит фикстуры, затем проходит с абсолютным config и package root;
2. structural-тест сначала обнаруживает tracked `outDir` в четырёх integration-файлах, затем подтверждает временный output; hash gate доказывает отсутствие мутаций полного suite;
3. D9-тесты с CRLF и одиночным CR сначала расходятся с LF, затем требуют глубокого равенства полного `D9Index`;
4. D8-тест с `not-a-date` сначала ошибочно проходит, затем отклоняется именно `format`-валидатором.

Финальные ворота:

- целевые тесты четырёх исправлений;
- полный `vitest run` из каталога пакета;
- полный запуск из системной временной директории командой, эквивалентной:

```powershell
$pkg = (Resolve-Path 'C:\tmp\hi-flow-arch-audit-js-esm\hi_flow\skills\arch-audit').Path
Set-Location $env:TEMP
& "$pkg\node_modules\.bin\vitest.CMD" run --config "$pkg\vitest.config.ts"
```

  Реальный абсолютный `$pkg` берётся из текущего worktree; команда не полагается на root-level `node_modules`.
- `tsc --noEmit`;
- `git diff --check`;
- hash-сравнение точного baseline tracked-отчётов до/после suite;
- `git check-attr` и байтовая LF-проверка Markdown-фикстур;
- отсутствие AJV warning в полном выводе;
- независимое ревью без незакрытых Critical/Important.

## Не входит в работу

- изменение D8 schema version или структуры отчёта;
- автоматическая команда обновления snapshot-файлов;
- реорганизация всего тестового дерева;
- устранение других Active Issues ArchAudit;
- добавление lint/CI baseline.

## Риски и ограничения

- `ajv-formats` — новая небольшая runtime-зависимость; она устанавливается стандартным `npm install` с обновлением lock-файла.
- Явный корень Vitest не заменяет абсолютные fixture paths: оба слоя нужны, потому что discovery и файловый ввод — разные границы.
- Очистка временных каталогов выполняется только для директорий, созданных самим тестом.
- Ошибка cleanup является ошибкой теста и остаётся видимой; helper не пытается маскировать её исходным исключением callback.

## Критерии готовности

- запуск полного suite из системной временной директории не зависит от текущего каталога;
- SHA-256 baseline всех tracked integration-отчётов полностью совпадает до и после suite;
- LF, CRLF и одиночный CR дают глубоко идентичный `D9Index`;
- `not-a-date` отклоняется с AJV `keyword: format` на `/metadata/audit_timestamp`, валидный RFC 3339 проходит, warning отсутствует;
- `.gitattributes` реально применяет `text eol=lf` ко всем Markdown-фикстурам;
- полный пакет зелёный, рабочее дерево после него содержит только намеренные изменения.
