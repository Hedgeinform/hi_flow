## 1. Назначение и scope

### Что делает skill

Per-feature **архитектурная дизайн-спека** Phase 2. **Мост между Phase 1 (поведенческая feature-spec) и Phase 3 (реализация через `superpowers:writing-plans`).** На входе — подписанная feature-spec фичи + снимок текущей архитектуры от arch-audit. На выходе — технический design-spec, достаточно детальный, чтобы writing-plans написал по нему план реализации без переоткрытия архитектурных решений.

**Позиция в pipeline.** В чистом superpowers роль технического design doc (что и как строить) играет ad-hoc brainstorming перед writing-plans. hi_flow заменяет это структурированной цепочкой: feature-spec (что, продуктово) → arch-spec (как, архитектурно) → writing-plans (план реализации). То есть arch-spec — **полноценная техническая дизайн-спека фичи**, наш способ произвести design doc: архитектурно-строгий и выводимый из продукта, в отличие от ad-hoc brainstorming, на который non-engineer оператор положиться не может. Это не нарушает D14: writing-plans (методология реализации) остаётся за superpowers; arch-spec даёт ЧТО (архитектура фичи), writing-plans — план КАК это реализовать.

**Что скилл делает — несколько задач, ни одна не единственная определяющая:**

1. **Выводит архитектуру из продукта** — арх-решения следуют из feature-spec, не изобретаются параллельно.
2. **Проектирует устройство фичи** — модули, контракты между ними, схема данных, точки встраивания.
3. **Проверяет встраивание в существующую архитектуру** — не плодит ли фича циклы, не размывает ли границы, не превращает ли соседний модуль в God object, не тянет ли зависимости не в ту сторону. Профилактика арх-нарушений в момент добавления фичи (операционализация «prophylactic» из D7 — предотвратить деградацию, чтобы потом не звать arch-redesign лечить).
4. **Декларирует инварианты** — для правильной реализации (профилактика) и защиты от регрессии при эволюции.

Профилактика встраивания (3) — важная задача, но **одна из четырёх**, не единственная суть. На чистом поле (первая фича) задача (3) неприменима, но скилл полноценно делает (1)(2)(4) — производит дизайн-спеку. Поэтому скилл нужен **всегда**, а не только при риске деградации: его основная работа — дать writing-plans полноценный архитектурный дизайн фичи; проверка встраивания добавляется, когда есть существующая архитектура.

### Что НЕ делает (явно out of scope)

- **Не пере-сканирует граф проекта.** Берёт готовый D8-снимок от arch-audit и рассуждает только о **дельте своей фичи**. Не дублирует машинерию аудита (граница с arch-audit: аудит смотрит на весь существующий граф, arch-spec накладывает фичу на готовую карту).
- **Не пишет в ARCHITECTURE.md** и не знает про скилл `architecture` / `living-architecture` (decoupling, OQ6 — см. §11).
- **Не строит enforcement.** Декларирует инварианты и указывает механизм проверки; сами проверки строят другие: rules-patch применяет оператор через arch-audit (D11), тесты пишет writing-plans, gate'ы настраивает L3 hygiene.
- **Не дублирует методологию реализации** (TDD, plan execution, code review) — это superpowers (D14).
- **Не сохраняет лог рассуждений** в выходном документе (отвергнутые варианты, история эскалаций). Спека = результат, не процесс. (Если эмпирически окажется нужным — отдельный артефакт в будущем, сейчас не проектируем.)
- **Не делает семантический анализ всего кода** — суждения LLM ограничены дельтой фичи, не всем проектом.

### Value proposition

arch-spec — **полноценная архитектурная дизайн-спека фичи, мост от продукта к реализации.** Превращает feature-spec в технический design так, чтобы writing-plans написал план без переоткрытия архитектуры. При этом гарантирует: (а) арх-решения выведены из продукта, не изобретены; (б) фича реализуется правильно с первого раза (инварианты доходят до writing-plans); (в) её добавление не деградирует существующую архитектуру (профилактика через анализ встраивания); (г) реализатор не дёргает оператора по мелочам (полнота спеки минимизирует эскалации). Профилактика арх-долга (в) — одна из этих гарантий, не единственная.

### Стратегическая позиция

Downstream двух upstream'ов: **feature-spec** (контракт фичи — независимый вход) + **arch-audit** (D8-снимок текущей архитектуры — независимый вход). Upstream одного consumer'а: **writing-plans**. Также порождает: rules-patch (consumer — arch-audit, D11) + backlog-кандидаты (consumer — product-backlog).

## 2. Вход и предусловия

### Входные артефакты

| Артефакт | Обязателен | Роль |
|---|---|---|
| `feature-spec.md` фичи | **Да** | Контракт фичи. Источник готовых арх-решений (схема, политики, точки эмиссии уже там) + продуктовое обоснование решений |
| `audit-report.json` (D8) | Условно (см. три ситуации) | Снимок текущей архитектуры — для блока C (встраивание) |
| `ARCHITECTURE.md` (Module Map) | Если есть | Карта существующих модулей, границ, известных проблем |
| `hi_flow/references/architectural-principles.md` (D9) | Да (read-only) | Каталог принципов для проверки дельты + ссылки в fitness-инвариантах |

### Три ситуации по аудиту (дифференциация — не копия arch-redesign)

Роль аудита у arch-spec **уже**, чем у arch-redesign: аудит нужен только для блока C (встраивание); блоки A/B/D/E выводятся из feature-spec и от аудита не зависят. Поэтому жёсткость другая.

Скилл сначала определяет режим по Module Map + git-истории:

| Ситуация | Поведение |
|---|---|
| **Чистое поле** (первая фича, кода нет / Module Map пуст) | Блок C легитимно неприменим. Работаем без аудита, помечаем «чистое поле». Это режим первого боевого прогона (audit-фича REH ERP) |
| **Обжитой код + свежий аудит** | Полный режим. Блок C работает |
| **Обжитой код + аудита нет / устарел** | **Громкий сигнал, не молчаливая деградация** (принцип 5). «Код есть, снимка аудита нет/устарел. Блок C — главная ценность — без снимка работает вслепую. Варианты: (а) прогнать arch-audit [рекомендуется]; (б) продолжить без анализа встраивания осознанно — блок C помечается “пропущен, нет снимка”, с логом причины» |

### Freshness check (когда снимок есть)

Идентично arch-redesign — консистентность семейства:
- Сравнить `audit_sha` из metadata снимка с `git rev-parse HEAD`.
- Совпало → молча работаем.
- Разошлось → мягкий вопрос: «аудит на коммите X, HEAD Y, прошло N коммитов. Перегнать arch-audit или продолжить?» Оператор решает, причина в лог.
- Не git → freshness пропущен с предупреждением.

### Schema check (когда снимок есть)

Снимок есть, но битый (нет reason-полей, нет `audit_sha`, нет ссылок на D9) → блок C на мусоре строить нельзя, жёсткий отказ **только для блока C** (остальное работает). Идентично arch-redesign schema check, но scope ограничен блоком C.

## 3. Flow (не линеен)

```
(feature-spec  ∥  свежий снимок arch-audit)   ← оба независимы по данным
                    │
              self-assessment (skip / direct / brainstorm)
                    │
                arch-spec
                    │
          ┌─────────┴──────────┐
     садится чисто         не садится (окружение кривое)
          │                     │
     writing-plans      ┌────────┼─────────┐
     (план реализации) упростить отрефакторить принять
                        продукт   окружение  долг
                        (назад на (arch-redesign (Known Drift,
                        feature-   перед фичей)   осознанно)
                        spec)
```

arch-spec — не только проектировщик, но и **развилка маршрута**: может завернуть назад на feature-spec (упростить продукт) или вбок на arch-redesign (почистить окружение), прежде чем пускать фичу в реализацию (Phase 2 → Phase 1 feedback loop из handoff 2026-04-27).

Очерёдность feature-spec ↔ arch-audit между собой **не важна** (они независимы по данным: feature-spec про продукт, audit про существующий код). Важно: оба готовы к моменту arch-spec, снимок свежий.

## 4. Probing taxonomy

### Принцип выведения

1. **Категории = архитектурные решения, которые надо принять**, чтобы заполнить блоки выходного документа. Не абстрактный список «-ilities» из учебника.
2. **Скилл сначала ИЗВЛЕКАЕТ готовые решения из feature-spec, копает только архитектурный слой.** Критично: иначе скилл переспрашивает то, на что feature-spec ответил. Первый шаг сессии — не «задать вопросы», а «прочитать feature-spec и вытащить готовые арх-решения» (схема данных, точки эмиссии, политики уже там), и только потом вопросы по дырам.

### Заземление на канон (deep-research 2026-05-31)

Floor + ceiling структура канонична: arc42 «pick only what matters», ATAM «importance × risk» (неравномерное распределение усилий). Floor статически детектируем (ArchUnit / dependency-cruiser / import-linter проверяют ровно это; Building Evolutionary Architectures классифицирует как static/automated fitness functions). Каждая категория обязана давать проверяемое решение/инвариант, не абстрактный ярлык (ATAM отвергает «должно быть модифицируемым» как непроверяемое). Top-3 дисциплина: probe только релевантные качества, не все.

**Caveat (честный):** все каноны системные; per-feature framing + сам mandatory floor — наша обоснованная дистилляция, не прямое предписание источников.

### Floor — всегда, статически детектируемо (структурная основа)

1. **Разбиение на модули** — по каким ответственностям бьётся фича.
2. **Зависимости и границы** — связи между модулями фичи и с существующими; направление, отсутствие циклов, чей интерфейс.
3. **Данные и состояние** — схема, где живёт состояние, единый владелец (SSoT). *Частично semantic: «кто владелец» — суждение, вне static-scope D9.*
4. **Точки встраивания** — где фича цепляется к существующему коду.

### Ceiling — по триггеру, top-3 релевантных (runtime/semantic)

5. **Режимы сбоев и устойчивость** — триггер: внешняя зависимость / error-prone путь.
6. **Нагрузка и масштаб** — триггер: растущий объём данных / горячий путь.
7. **Безопасность и границы данных** — триггер: секреты / PII / контроль доступа.
8. **Трансляционная граница (Anti-Corruption Layer)** — триггер: семантический mismatch с чужой/legacy подсистемой. Анти-триггер: нет значимой семантической разницы. *Добавлено из research — отличается от «точек встраивания»: точки = ГДЕ цепляемся; трансляционная граница = НУЖЕН ли адаптер, чтобы чужая семантика не протекла в модель фичи (DDD/Azure ACL).*
9. **Согласованность и идемпотентность** — триггер: запись в несколько хранилищ / повторяемые операции / конкурентная запись.
10. **Эволюция контракта/схемы** — триггер: фича меняет shared контракт/схему, который кто-то потребляет.

### Сквозные проверки (на каждое решение)

- **Дельта-влияние на граф** — привязка к конкретному модулю/ребру (Risk Storming: риск локализуется к элементу, на нескольких уровнях гранулярности).
- **Выводимость из продукта** — гейт «это арх или продукт?»: non-domain design consideration + влияет на структуру + критично для успеха (three-criteria filter Richards & Ford). Отсекает переоткрытие продуктовых решений feature-spec.
- **Противоречия** — как в feature-spec.
- **Наследуемые cross-cutting политики** — какие системные политики (логирование/observability, error-handling, auth, secret-mgmt) фича инстанцирует (arc42 §8). Сюда ложится **observability** — через глобальный принцип 5 (молчаливый fallback запрещён → видимое логирование) применяется всегда, но как наследуемая политика, не отдельная структурная floor-категория.

### Производная

- **Fitness-инварианты** — не отдельный «допрос», а сборка из решений floor + ceiling + наследуемых политик (см. §7).

### Закрывающий шаг

- **Инвентаризация пределов** (двухуровневая, см. §6) — заменяет архитектурный premortem. Premortem уместен для неопределённости (продукт), а код — формальная область: границы выводятся, не воображаются.

### Adaptive depth + эмпирическая доводка

Глубина по типу фичи (D3 floor+ceiling): копаем там, где у фичи вес, пропускаем пустое. **Полнота taxonomy проверяется эмпирически на первом боевом прогоне** (audit-фича) — состав может быть скорректирован по результату.

### Сверка с D9 (консистентно)

Floor 1-2 ↔ D9 группы 1-2 (SOLID на модулях + Package Principles) + structural classics; floor 4 и ceiling 8 ↔ группа 3 (Hexagonal/Boundaries). Прогон дельты по принципам (блок C) = прогон по D9. Drift: SSoT (floor 3) — semantic, вне static-scope D9 (помечено выше).

## 5. Блок C — анализ встраивания (ядро скилла)

### Ключевой инсайт: дельта — прогноз, не измерение

Фичи ещё нет в коде. «Влияние на архитектуру» нельзя измерить — только предсказать: «если реализовать как спроектировано в блоке B, граф изменится так». Следствия:
- **Порядок блоков жёсткий: B → C.** Нельзя предсказать влияние, не спроектировав устройство.
- arch-spec строит **гипотетический граф** = текущий (из снимка) + узлы/рёбра фичи (из блока B).

### Модульный уровень (не файловый)

Граница реализуемости: на уровне **файлов** задача почти нереализуема (предсказывать несуществующие импорты). На уровне **модулей** — управляема: текущий граф модулей уже в снимке (`per_module` метрики + `dep_graph`), граф фичи модульный = декларация блока B, склейка тривиальна (десятки узлов, не сотни).

**Операциональное определение «модуля»:** модули фичи декларируются в **той же гранулярности, что `per_module` в D8-снимке** (у arch-audit это директория-модуль уровня `src/<module>/`). Это обязательно для склейки: если новый модуль фичи объявлен в иной гранулярности, чем узлы снимка, наложение ломается. Правило: новый модуль фичи = новая директория-модуль того же уровня; зависимость «модуль фичи → существующий модуль» = ребро между узлами этой гранулярности. Имплементатор не гадает о соответствии единиц.

### Гибрид: код для структуры, LLM для семантики (принцип 2)

| Проверка | Чем | Тип |
|---|---|---|
| Новые циклы | Обход графа | Код |
| Направление зависимостей (SDP) | Пересчёт нестабильности с новыми рёбрами | Код |
| Рост связности (NCCD) | Пересчёт, сравнение с текущим | Код |
| God object / hub-пороги | Превышение порога связей | Код |
| Нарушение существующего fitness-правила | Проверка нового ребра против списка запретов | Код |
| Границы на новых стыках | «Лезет ли мимо интерфейса» — суждение о смысле | LLM |
| Дублирование ответственности | Граф не видит, что фича делает то же | LLM |
| Классификация исхода нарушения | Переделать фичу / рефакторить окружение / принять долг | LLM |

**Вход/выход LLM-проверок (чтобы не было «здесь магия»):** на вход LLM-проверки подаётся — контракты модулей фичи из блока B (ответственность, что экспортирует, от чего зависит) + reason-поля и метки соседних узлов из D8-снимка (что эти модули собой представляют). На выход — вердикт, привязанный к конкретному ребру/узлу гипотетического графа (как в Risk Storming: нарушение локализуется к элементу), в той же форме, что finding arch-audit (тип нарушения + reason + source/target). Это держит LLM-часть в том же формате, что формальную, и делает её проверяемой оператором.

**SSoT-требование (принцип 4) — с важной оговоркой про реальное состояние arch-audit:**

- **Формулы метрик** (NCCD — `helpers/compute-nccd.ts` уже чистая функция; Ca/Ce/I — если выделяемы как чистые функции) **импортируются** из arch-audit, не переписываются. Вот тут SSoT: один показатель — одна формула.
- **Цикл-детекция — пишется заново, НЕ импортируется.** В arch-audit циклы приходят из вывода внешнего depcruise (`helpers/parse-depcruise-output.ts`, поле `v.cycle`), а не из внутреннего алгоритма. depcruise сканирует реальные файлы — на гипотетическом графе фичи (кода ещё нет) его запустить нельзя. Поэтому для дельта-графа нужен **новый алгоритм обхода** над `dep_graph` снимка + рёбрами фичи. Это новая разработка, а не extraction.

Конкретное требование к реализации: **создать shared graph-core модуль** — чистые формулы метрик (вынести из arch-audit, где они есть) + новый graph-traversal для циклов/достижимости на декларативном графе. Импортируется обоими скиллами. Это **отдельная upstream-задача** (принцип 10: downstream arch-spec impl не стартует до её мёржа), а не пункт «между делом» — затрагивает работающий arch-audit (нужен прогон его тестов после рефактора). См. §16, §18.

Честная цена модульного уровня: границы ловятся слабее, чем файловый аудит (на файлах depcruise видит точные импорты). Часть «нарушение известного правила» — формально, часть «новый подозрительный стык» — LLM.

### Процесс (4 шага — цикл, не однопроходная проверка)

1. **Построить дельту графа** — наложить модули/рёбра фичи (блок B) на снимок.
2. **Прогнать граф будущего по принципам** — циклы / границы / направление / God object / дублирование.
3. **Классифицировать каждое предсказанное нарушение:**
   - избегается переделкой фичи → назад в блок B (предпочтительно);
   - неизбежно без рефакторинга окружения → сигнал наверх;
   - осознанно принимается → Known Drift (редко, под расписку).
4. **Сигнал наверх (если шаг 3 упёрся)** — упростить продукт / arch-redesign на окружение / принять долг.

## 6. Границы работоспособности (двухуровневая инвентаризация)

Заменяет premortem. Двухуровневость — чтобы не считать пределы там, где их нет (формат `actor_label` не «упрётся»; миллион запросов/сек у ERP недостижим).

**Уровень 1 — дешёвый триаж.** По каждому значимому решению быстрое суждение «да/нет»: задевает ли ось с потенциальным пределом? Сигналы риска: растущий объём данных / горячий путь / внешняя зависимость / конкурентность / ресурсный лимит. Нет сигнала → «предел не применим», дальше не копаем. **Мгновенный отсев большинства решений.**

**Уровень 2 — только для прошедших фильтр.** Зафиксировать границу работоспособности.

**Распределение по артефактам:**

| Кусок | Куда | Почему |
|---|---|---|
| Решение + граница-допущение («без партиционирования, валидно до 1 года») | **В спеку**, одна строка | Рамка, в которой writing-plans пишет план. Не даёт over-engineer'ить |
| Триггер мониторинга (при > Y приближаемся к пределу) | **product-backlog** | Текущей реализации не нужен |
| Следующая ступень (партиционирование / шардинг) | **product-backlog** | Будущая работа, не эта фича |

writing-plans видит только границу-допущение (рамки). «Что делать за границей» — отложенная работа в формате «при достижении Z разработать W».

Стресс-тест на audit: триаж пропускает на уровень 2 `audit_log` (растёт), эмиссию (горячий путь), overflow в storage (внешняя зависимость); отсекает формат `actor_label`, конвенции snapshot.

## 7. Fitness-инварианты

### Первичный потребитель — реализация, не аудит

Фундаментальный приоритет: если инварианты уходят только в arch-audit, профилактика превращается в лечение «на день позже» (код пишется не зная инвариантов → аудит ловит постфактум). Профилактика = инвариант доходит до writing-plans **до** написания кода.

| Потребитель | Когда | Роль |
|---|---|---|
| **writing-plans → реализация** | сейчас, при написании фичи | **профилактика** — код сразу по инвариантам. Первичный |
| **arch-audit** | потом, при эволюции | **страховка + анти-регрессия** — ловит нарушения мимо плана и поломку старых инвариантов новыми фичами. Вторичный |

### Классификация по механизму проверки

| Тип | Что это | Куда |
|---|---|---|
| **1. Граф-формализуемые** | Зависимости модулей: кто кому имеет право импортировать | В спеку **+ дополнительно** rules-patch для arch-audit (авто-проверка) |
| **2. Код-содержание / схема** | SQL-операции, содержимое payload, миграция БД | В спеку как инвариант + указанный механизм (тест / lint / проверка миграции) → строит writing-plans |
| **3. Динамические** | Latency, error rates | Текстовое требование к мониторингу, без enforcement (вне scope сейчас, как D9 static-only) |

Что граф-формализуемо — решает LLM при формулировке (про импорты или про SQL). Все типы → в дизайн-спеку как явные инварианты с указанием механизма (первичный путь, профилактика). Граф-часть **дополнительно** экспортируется как rules-patch (вторичный путь, анти-регрессия).

### Формат rules-patch = как arch-redesign (D11)

Потребитель один (`arch-audit apply-patch`), контракт D11 общий. arch-spec **не изобретает** свой формат: YAML правил зависимостей, каждое с `name` / `severity` / `from`/`to` / обязательной ссылкой на принцип D9. Apply — explicit действие оператора (не авто).

### Источник

CC (cross-cutting policies) и P- (reusable sub-policies) из feature-spec — готовые кандидаты в fitness-инварианты, arch-spec их формализует. Плюс решения блока B.

Стресс-тест на audit: P-EMISSION-BOTTLENECK → граф-правило «импорт `audit-emitter` только из 3 bottleneck-модулей» (тип 1, rules-patch); CC1 immutability «нет UPDATE/DELETE на audit_log» → тип 2 (инвариант + тест); F10.1 indexes → тип 2 (проверка миграции).

### Накопительный цикл (D11)

Фича вносит свои граф-правила в накопительный rules-файл → будущие прогоны arch-spec (блок C, «нарушение известного правила») и arch-audit их учитывают. Каждая фича укрепляет защиту следующей.

## 8. Граф связей (визуализация)

### Математический граф ≠ визуализация

| | Математический | Визуализация (Mermaid) |
|---|---|---|
| Зачем | Машина считает дельту (блок C) | Человек видит стык фичи |
| Лимиты | Нет (данные) | Есть (Mermaid плывёт после ~30 узлов) |
| Показывает всё? | Да | **Нет** |

Прецедент лимита в семействе: product-spec D15 ушёл с function-level на module-level Mermaid из-за нерендеринга 30+ нод.

### Решение: ego-граф фичи

Визуализация показывает **не весь граф, а окрестность фичи** — сама фича + модули, с которыми граничит напрямую (радиус 1-2 связи). Обычно 5-10 узлов — Mermaid справляется. Подсветка дельты: новые модули/связи — цветом, существующие соседи — серым. Полный граф продукта — забота arch-audit (системный Mermaid) и Module Map, arch-spec не дублирует.

Деградация при большой окрестности (фича задевает 20 модулей — редкость): группировка соседей по слоям в subgraph / радиус 1 / текст + частичный граф.

## 9. Output document (`arch-spec.md`)

### Чистота — потребитель writing-plans (агент), не оператор

Тело спеки должно быть **чище feature-spec** (который читает оператор): только результат, без истории развилок. Решения как факты + одна строка rationale инварианта (зачем правило, чтобы не нарушить) — но НЕ история выбора (что отвергли). Эскалация остаётся в сессии, в документ попадает только принятое решение.

### Блоки анализа A-E → секции документа

Probing taxonomy (§4) и анализ организованы по пяти блокам; 10 секций документа — их проекция:

| Блок | Что | Секции документа |
|---|---|---|
| **A. Вход и исходное состояние** | Что прочитано, во что встраиваемся | 1 (шапка), 3 (цель/границы), 4 (исходное состояние) |
| **B. Целевое устройство фичи** | Модули, контракты, данные, точки встраивания | 5 |
| **C. Влияние на архитектуру** | Дельта графа, деградация, сигнал наверх | 6 |
| **D. Защита реализации от отклонения** | Fitness-инварианты + делегированные развилки (+ эскалация как механика сессии, §10 спеки) | 7, 10 |
| **E. Подача для человека** | Сводка + визуализация | 2 (Summary), 9 (граф) |

Границы работоспособности (секция 7, §6) — часть блока D (защита). Эскалация (§10) — механика сессии, не отдельная секция документа.

### Структура — 10 секций

1. **Шапка** — ссылка на feature-spec, снимок аудита (sha + свежесть), дата, статус (draft/signed), режим (чистое поле / обжитой).
2. **Summary** — компактная сводка арх-принципов фичи. Якорь для оператора, plain language (P1). Обязателен (handoff 2026-04-27).
3. **Цель и границы** — что решено на арх-уровне, что делегировано реализации, что в бэклог.
4. **Исходное состояние** (блок A) — что прочитано + что важно для встраивания (соседи, границы, известные проблемы). Чистое поле → пометка.
5. **Целевое устройство фичи** (блок B) — floor: модули / контракты / данные+SSoT / точки встраивания; + условные ceiling-секции по триггеру (только задетые).
6. **Влияние на архитектуру** (блок C) — дельта графа / проверка деградации по принципам / сигнал наверх.
7. **Границы работоспособности** — инвентаризация пределов (граница-допущение; триггеры+ступени → бэклог).
8. **Fitness-инварианты** — список с классификацией по механизму. Граф-часть дополнительно как rules-patch файл рядом.
9. **Граф связей** — Mermaid, окрестность фичи с подсветкой дельты.
10. **Делегировано реализации** — явные развилки для writing-plans (с инструкцией «выбери, видя код, учти ограничение»).

**Нет секции «открытые вопросы».** Открытый вопрос либо резолвится в сессии (→ факт в спеке), либо откладывается (→ product-backlog). Третьего нет — отдельной секции внутри спеки не нужно, иначе deferred-элементы размазываются по N файлам (урок feature-spec feedback §1).

### Sample dialog — опционален

Для фич с нетривиальным многомодульным потоком — краткая «трассировка ключевого потока» (для audit: действие → bottleneck → store → overflow/retention → UI). Показывает архитектуру в действии, дополняя статический граф динамикой. Для простых фич избыточен — пропускается.

### Output path

`<project>/docs/specs/YYYY-MM-DD-<feature-slug>-arch-spec.md` + rules-patch файл рядом (`<feature-slug>-rules-patch.yaml` или аналог). Configurable.

## 10. Эскалация

**Эскалация — механика сессии, не раздел документа.** Дисциплина «когда дёргать оператора» живёт в SKILL.md (как arch-spec ведёт себя в сессии), не в выходной спеке.

- **Предвиденное** (развилки, видимые к закрытию): решение либо принято (→ факт в спеке), либо делегировано реализации (→ секция 10, с инструкцией). Наследует severity-разметку из feature-spec Open items.
- **Непредвиденное** (всплыло в реализации): общий критерий objective-impact (UX / безопасность / стоимость / поддержка → эскалация с обоснованием; иначе → реализатор сам). arch-spec не переизобретает критерий — он в P6 и superpowers. Не вписываем в каждую спеку.

**Цель — минимизировать эскалации полнотой спеки.** Эскалация в реализации = признак, что спека чего-то не закрыла. Стремимся к нулю (метрика качества спеки).

## 11. Связь с living-architecture (OQ5) + decoupling (OQ6)

### Вариант 2 — полный decoupling

arch-spec **decoupled**: выдаёт спеку с решениями как фактами, в ARCHITECTURE.md **не пишет**, про архитектурный скилл **не знает**. Связь — только через артефакт (спеку), слабая. Living-architecture (если установлен) сам извлекает решения своим event detection; нет (market-ready) → оператор вручную / аналог в плагине. Зависимости от неизвестного скилла нет (стабильнее для distribution, чем delta-блок-интерфейс).

### Напряжение фоновой фиксации — зона living-architecture

Архитектурный скилл работает фоном всю сессию и по дизайну детектит решения по ходу. Напряжение: решения фичи в процессе arch-spec могут переворачиваться → преждевременная фиксация сырого.

Разрешение (асимметрия, сохраняет decoupling):
- **arch-spec** в неведении про фоновый скилл.
- **living-architecture** (мета-скилл, ведущий документ) знает: когда активен design-скилл семейства, фиксация **проектных** решений откладывается до закрытия артефакта (signed).

**Это контракт для будущей сессии living-architecture (D20), здесь не углубляем.** Обязательство arch-spec: дать чёткую точку «спека signed = решения финальны», чтобы потребитель имел стабильный момент подхвата.

### Разведение уровней

| Что | Куда |
|---|---|
| Решения про **фичу** проекта X (как audit разбита на модули) | arch-spec.md → потом в ARCHITECTURE.md **проекта X** |
| Решения про **сам hi_flow** (дизайн arch-spec) | ARCHITECTURE.md **hi_flow** — мета-сессия, через operator-personal architecture |

## 12. Closure backlog-sync

Отложенное (следующие арх-ступени из §6, открытые вопросы для других/высших уровней) идёт в **product-backlog** (внешний файл), не в спеку. При закрытии arch-spec собирает отложенное и **предлагает занести в product-backlog** (patch на утверждение, не молча).

**Механизм согласуется с feature-spec backlog-integration design** (`docs/superpowers/specs/2026-05-29-hi_flow-feature-spec-backlog-integration-design.md`): harvest по структурным якорям → detect `*backlog*.md` → idempotency check по `Originating analysis` → build patch + approval → No-backlog flow (создать из `product-backlog-template.md`). arch-spec — bottom-up контрибьютор backlog'а (read format + append + create-if-missing), как и feature-spec. Формат записи — `product-backlog-template.md` (single source).

**ВНИМАНИЕ — cross-branch:** артефакт backlog-integration design лежит в **main**-ветке репозитория; текущий worktree (`silly-shtern`) ответвлён раньше 2026-05-29 и его НЕ содержит. При impl arch-spec и при мерже этой ветки — **обязательная сверка** с актуальным состоянием backlog-integration в main (путь/имя/решения могли измениться). Не опираться на память этой сессии — перечитать артефакт из main.

**Зависимость реализации:** closure backlog-sync arch-spec требует, чтобы общесемейный backlog-integration механизм был реализован (на момент 2026-05-31 — design в main, impl pending). arch-spec impl координируется с ним (общий closure-механизм, не два разных) — явная upstream-зависимость (принцип 10).

## 13. Self-assessment (skip / direct / brainstorm)

Тот же паттерн, что feature-spec. Факторы под арх-уровень:
- **direct** (скилл предлагает пропустить brainstorm, выдать спеку напрямую): простая фича, 1-2 модуля, чистое поле или зелёный аудит, мало ceiling-триггеров.
- **brainstorm** (conservative default): много затронутых модулей, обжитой код с известными проблемами рядом, несколько ceiling-триггеров, сигналы возможной деградации.
- **skip**: оператор явно отказывается (берёт ответственность).

Формат self-assessment proposal — как у feature-spec (информативный, редактируемый).

## 14. Self-review checklist (изолированный субагент, P5)

Изолированный субагент (без истории сессии) проверяет выходную спеку:
- floor-категории (1-4) покрыты;
- ceiling-секции присутствуют для всех задетых триггеров;
- fitness-инварианты actionable (проверяемы), не абстрактны; у каждого указан механизм + ссылка на D9;
- дельта (блок C) проверена против принципов; нарушения классифицированы;
- решения выводимы из feature-spec (нет «изобретённой» архитектуры без продуктового корня);
- в спеке нет мета/истории рассуждений; решения как факты + строка rationale;
- границы работоспособности зафиксированы для прошедших триаж; триггеры+ступени ушли в бэклог, не в спеку;
- Summary — plain language (P1);
- rules-patch валиден (формат D11, ссылки на принципы существуют).

Safe-to-autofix → применить inline; human-required → флаг оператору.
