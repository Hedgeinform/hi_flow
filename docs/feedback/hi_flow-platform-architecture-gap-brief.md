# Скилл-бриф: зазор «локально оптимальная архитектура vs глобальная архитектура проекта» (intentional architecture for shared capabilities)

**Сессия:** 2026-06-03, arch-spec по фиче `comm` (REH ERP, итерация 1).
**Затрагивает скиллы:** `hi_flow:arch-spec`, `hi_flow:product-spec` (декомпозиция итерации), потенциально новый скилл / шаг.
**Повод:** при проектировании comm обнаружилось, что **первая** фича, которой нужна сквозная инфраструктурная способность, определяет её узко под себя — потому что arch-spec выводит дизайн из *текущего снимка* проекта (block C — «деградирует ли граф сейчас»), а не из roadmap. Когда вторая фича приходит, появляется дублирование, которое классифицируется как Known Drift и едет в backlog. Это работает (план «emergent + дёшево унифицируем потом» — валидная ставка), но **системно делает первую фичу архитектурным владельцем сквозных способностей по умолчанию**, без явного шага «здесь общая платформенная абстракция».

---

## Суть: emergent architecture vs intentional architecture (architectural runway)

hi_flow по конструкции — **emergent-per-feature**:
- `product-spec` декомпозирует продукт на фичи.
- `feature-spec` фиксирует продуктовые развилки по фиче.
- `arch-spec` проектирует архитектуру **под одну фичу** + проверяет интеграцию в *текущий* граф (block C).

Это даёт хороший уровень детализации для writing-plans и быстрый поток. Цена ставки — **сквозные инфраструктурные способности структурно бездомны в основном потоке**: первая фича, которой они понадобились, узко их определяет; вторая фича сталкивается с обобщением задним числом.

**Зазор затрагивает не «всю архитектуру вообще», а конкретный класс — shared capabilities:**
object storage, каналы коммуникации, scheduler/cron, auth, шина событий, observability, файловое хранилище, idempotency-key инфра, secret-manager. Доменные фичи этим не болеют (они естественно вертикальные слайсы) — болеет только инфраструктурный слой.

---

## Реальный кейс (живой пример, не гипотетика)

В REH ERP это уже сработало **дважды подряд**, и второй раз — на глазах:

1. **audit-фича (первая)** ввела `audit-overflow` — порт+адаптер для overflow JSON-payload'ов. Решение **локально оптимальное** (узкий порт под единственного потребителя — это правильный YAGNI на момент arch-spec audit). Глобально — нет: это был первый потребитель концепта «blob за S3-портом», но способность сквозная (нужна вложениям comm, документам контрактов, медиа tickets — это всё в roadmap, видно из backlog `INFRA-X-deploy-bindings` и D7).

2. **comm-фича (вторая, текущая)** ввела общий порт `object-storage` (bytes, generic key) — то есть **переоткрыла ту же концептуальную способность** уже как общую. Возникло дублирование (два порта-сиблинга: специализированный audit + общий object-storage). Унификация осознанно отложена в backlog (`F-comm-X-object-storage-unify`).

3. **Третий повтор готовится прямо сейчас в той же спеке.** comm также вводит порт `comm-channels` (каналы коммуникации). Этот порт **по природе сквозной** — его будут потреблять notif (доставка уведомлений), tickets (коммуникация с подрядчиком), portal (уведомления о сообщениях). То есть first-feature-defines-shared-port **готов повториться** — и предсказуем заранее.

**Не дефект arch-spec'ов отдельных фич** — каждая из них корректна локально. Дефект — **в потоке между ними**: нет шага, на котором сквозная способность опознаётся как сквозная *до* того, как первая фича её определит.

---

## Частичная страховка, которая уже есть в hi_flow (и почему её недостаточно)

Знание о сквозных способностях в системе **существовало** и до этой сессии:
- backlog флагал `INFRA-X-deploy-bindings` как project-level (object storage + scheduler).
- ARCHITECTURE.md держит D7 как «инфраструктурная константа итерации».
- У hi_flow есть `arch-redesign` — отдельный проход, специально под консолидацию emergent-долга.

**Чего недостаточно:**
- Знание лежит в backlog'е/архитектурных решениях как **флаги**, без **структурного владельца** (не спроектированный общий модуль-порт).
- `arch-redesign` — лечение пост-факт; он работает, когда долг уже накоплен. Профилактика дешевле — особенно когда сквозной характер способности **очевиден заранее** из roadmap (как с каналами в comm).

## Предлагаемые правки (по возрастанию веса — выбрать одну глубину)

### Вариант A — Минимум: ничего структурного

Полагаемся на `arch-redesign` как periodic consolidation pass. Платим reconciliation-cost задним числом каждый раз. Если поток короткий (1–2 фичи, использующих одну способность) — приемлемо; если длинный (object-storage в REH ERP будет потреблён 4–6 фичами) — дорого.

### Вариант B — Лёгкий: реестр платформенных способностей в ARCHITECTURE.md *(рекомендую)*

Добавить в `ARCHITECTURE.md` раздел типа **«Platform capabilities (shared ports)»** — лёгкий реестр сквозных способностей: имя порта, владелец (модуль, реализующий), потребители (фичи, использующие), статус (PLANNED / BUILT / PARTIAL). Заполняется при первом упоминании способности в любой фиче и обновляется по мере появления потребителей.

`arch-spec` тогда:
1. В block A (starting state) **читает** этот реестр и проверяет: «вводит ли моя фича сквозную способность? есть ли она уже в реестре?»
2. Если способность уже зарегистрирована → потребляем существующий порт (даже если он сейчас single-user — расширяем контракт осознанно).
3. Если фича вводит **новую** способность, которая выглядит сквозной (по roadmap / backlog) → реестр сразу её регистрирует как platform port (не как module of this feature), и arch-spec проектирует её под roadmap, а не только под себя.

Дёшево, без новых скиллов, ловит и object-storage (задним числом), и каналы (упреждающе). Хорошо ложится на существующий `architecture` skill (он уже владеет ARCHITECTURE.md).

### Вариант C — Тяжёлый: отдельный шаг «platform arch-spec» (или новый скилл `hi_flow:platform-spec`)

Перед/параллельно потоку фичевых arch-spec'ов — отдельная фаза, которая на основе product-spec'а и roadmap проектирует **сквозные способности целиком** (object-storage, channels, scheduler, auth, events). Фичевые arch-spec'и тогда не вводят shared ports, только потребляют их.

Преимущество: интенциональная архитектура для всего слоя сразу. Недостаток: возвращает элемент big-design-up-front, риск спекулятивного дизайна сквозных абстракций. Имеет смысл для крупных итераций с очевидно-сквозными способностями.

## Конкретные правки в скиллах под Вариант B

**В `architecture` skill (или его шаблоне `references/template.md`):**
- Добавить опциональный раздел `## Platform capabilities` со схемой записи (port name, owner module, consumers, status, spec-ref).
- В event detection — добавить триггер «фича вводит / потребляет сквозной порт» как архитектурное событие, требующее фиксации.

**В `hi_flow:arch-spec`:**
1. В блоке A (Starting state) — обязательный шаг: **прочитать platform capabilities registry** (если существует) и сверить со scope фичи.
2. Новая probe-категория в floor (или sub-floor) — **«Shared capability detection»**: вводит ли фича новый сквозной порт? Признаки сквозного характера: (а) backlog/roadmap упоминает других потребителей этой способности; (б) арх-снапшот / ARCHITECTURE.md флагует её как infrastructure constant; (в) способность ортогональна доменной логике фичи.
3. Если способность сквозная — в §5.1 явно фиксировать её **как platform port**, не как module-of-this-feature; в §3 (goal/boundaries) явно отделять «фича-локальные модули» от «platform ports, владельцем которых фича становится».
4. На closure — patch для platform capabilities registry (обновление реестра, не только backlog-sync).

**В `hi_flow:product-spec` (декомпозиция итерации):**
- При формировании bundle-файлов добавить опциональную секцию `## Shared capabilities surfaced by this feature` — какие сквозные способности фича упоминает (для информирования arch-spec'а).

## Приоритет

**Medium-High.** Не блокер (ставка emergent + arch-redesign работает), но **системный и накапливающийся**: каждая новая фича с инфраструктурной потребностью повторяет паттерн, и долг растёт линейно от размера roadmap. Особенно болезненно для проектов вроде REH ERP с 13 фичами в итерации, где сквозных способностей много (object-storage, channels, scheduler, auth, events).

Дешёвая профилактика на уровне реестра (Вариант B) экономит много arch-redesign-проходов и держит дизайн сквозного слоя осознанным с первого касания, не вторым.
