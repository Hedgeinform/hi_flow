# `hi_flow:product-spec` — Product-Level Decomposition Skill

Помоги оператору превратить продуктовую идею (или новую итерацию существующего продукта) в замороженный `product-spec.md` для одной отгружаемой итерации плюс живой `product-backlog.md`, который накапливает реестр функций между итерациями.

Скилл системно вытаскивает пользовательские (domain) функции и производные поддерживающие (enabler) функции через структурированный опрос на продуктовом языке (сверху вниз per D9), классифицирует стратегические развилки по тесту границ D5, управляет памятью между итерациями через бэклог с асимметричной дисциплиной размера записей.

## Out of scope

- Углубление в одну фичу — отдельный скилл `hi_flow:feature-spec`.
- Архитектурные решения — `hi_flow:arch-spec` (выводит архитектуру из готовых product-spec + feature-spec).
- Implementation planning — downstream feature work uses `hi_flow:implementation-plan` after `feature-spec` / `arch-spec`; Superpowers remains the recommended execution discipline, not the signed-spec planner.
- Auto-invoking следующей фазы — оператор инициирует переход вручную.

## Concepts

Базовые понятия, на которых стоит вся таксономия скилла. Если оператор работает с product-spec впервые — это словарь.

- **Trinity функций (D1).** Любая **функция** в продукте (атомарная capability per D2) — одна из трёх категорий: **пользовательская функция** (domain capability) даёт ценность юзеру напрямую (подача заявки, аутентификация, поиск); **поддерживающая функция** (enabler capability) обслуживает пользовательские, но содержит продуктовые развилки (управление учётками, RBAC, нотификации); **scaffolding** — чисто инженерные решения без продуктовых развилок (схема БД, deployment, CI/CD, бэкапы). Product-spec фиксирует только domain + enabler функции. Scaffolding — материал arch-spec, в product-spec не записывается.
- **Фича = группа функций, не уровень иерархии (D3).** Заголовок «### Фича: <Name>» в Section 4 — organisational метка для группировки родственных функций (Identity & Access, Communications, Sales Pipeline), а не уровень декомпозиции. Функции внутри фичи равноправны и связаны через DAG, не через вложенность. Фича не имеет собственной карточки и собственных полей. **D2 application:** atomicity gate D2 (атомарность / отгружаемость / переговариваемость) применяется к **функциям** — это decomposition gate для разбиения user jobs на атомарные единицы. Фича — grouping decision (Feature assignment Шага 4-5 / Sub-phase 1 Decomposition phase), не D2 gate; фича aggregate, не atomic; её independent shippability — emergent property её функций.
- **Terminology realignment (v0.6.1).** Предыдущие версии (v0.5.0 и ранее) использовали «module» в product-spec'е как concept для группировки функций; «feature» использовалось как synonym для «function» (atomic shippable unit). V0.6.1 realign terminology под industry standard (Lean UX / Scrum): **фича = aggregate of capabilities** (раньше — module/cluster), **функция = atomic capability** (раньше — function/feature в narrow sense). Module/cluster/feature концепты collapse в один — фича. F-* идентификаторы stable (структурно `F-<feature-slug>-<N>` parses естественно). Module Map в ARCHITECTURE.md — separate concept (code-modules, не product features), не trogaem.
- **DAG зависимостей (D4).** Связи `Зависит от` между функциями образуют направленный ациклический граф. Циклы — блокирующая ошибка целостности (см. Шаг 12.2). Порядок функций в backlog § Committed и при closure определяется топологическим сортом DAG'а: зависимости впереди зависимых.
- **Одна спека = одна итерация (D16).** Product-spec фиксирует ровно одну отгружаемую итерацию. Никаких «v2 parked в этой же спеке» или «v3 roadmap внизу». Будущие итерации живут в backlog'е (parked / deferred), а сама v2 — это отдельная новая product-spec через несколько месяцев. Latest wins (D18).

## Output

**Три типа артефактов:**

1. **product-spec.md** — замороженная запись процесса дизайна одной итерации. Default location: `<project>/docs/specs/YYYY-MM-DD-<product-slug>-<iteration-slug>-product-spec.md` (configurable per OQ15).

2. **product-backlog.md** — реестр функций, живущий через все итерации одного продукта. Default location: `<project>/docs/specs/<product-slug>-product-backlog.md` (configurable). Создаётся при первой итерации, обновляется в каждой последующей.

3. **Plan directory (v0.6.0)** — `<project>/docs/specs/<product-slug>-iteration-<N>-plan/`, содержит:
   - `roadmap.md` — план реализации (последовательность блоков функций с межблочными зависимостями).
   - `bundle-<feature-slug>.md` × N — контекстные пакеты для feature-spec сессий по каждой фиче.

   Создаётся в decomposition phase (см. раздел `## Decomposition phase` ниже) — closure-фаза после Шага 12. Может отсутствовать если оператор пометил спеку `decomposition: skipped` либо отложил generation.

## Anti-triggers (do NOT auto-activate)

- «спека на фичу X», «продуктовый дизайн фичи X» — это уровень feature-spec, не product-spec. Предложить правильный скилл, не запускать product-spec.
- «давай добавим функцию Y в продукт» — может быть либо feature-spec (если продукт уже существует), либо update mode product-spec (если меняем scope текущей итерации). Уточнить у оператора, не запускать автоматически.
- «спроектируй архитектуру для X» — территория arch-spec.
- «реализуй X», «напиши код» — фаза имплементации.
- «расскажи про методологию» — research / explanation, не запуск скилла.

## Mode

Скилл работает **только через brainstorm-путь**. Direct-путь не применяется на продуктовом уровне — продукт по природе многофункциональный, многосегментный, и вывод поддерживающих функций сверху вниз (D9) теряется при пропуске опроса.

**Никаких self-assessment предложений оператору про выбор пути** (direct vs brainstorm — D13 фиксирует brainstorm-only, выбора нет). Session setup нормально задаёт **операционные вопросы** про identification активной спеки / project root / slug'и — это не self-assessment, это setup, и без этих ответов скилл не может корректно стартовать.

Если оператор явно скажет «direct» / «без проб» — отказать со ссылкой на D13 и предложить короткий brainstorm: тривиальные шаги в простых продуктах пробегаются быстро, но проходятся все.

## Process flow

Сессия проходит через четыре фазы:

1. **Session setup** — определение активной спеки и загрузка бэклога. **В v0.6.0** добавлена detection branch «signed + no plan» — Entry B в decomposition phase (см. ниже).
2. **Operator dump** — открытое описание продукта оператором без структурирования.
3. **Agent probing** — двенадцатишаговая таксономия (см. ниже), последовательно. Сквозные проверки активны параллельно.
4. **Closure** — Шаги 4-11 → завершающий премортем (может породить дельту: новые функции / forks / scope cuts) → при наличии дельты — return в Шаги 4-6 для добавления → Шаг 12 (подтверждение scope + проверка целостности + миграция в бэклог при закрытии) + таблица открытых вопросов. Премортем **до** Шага 12, не после, иначе порождённые им функции не попадут в спеку. **В v0.6.0** Closure расширена: после Шага 12.4 запускается decomposition phase per Entry A gate (см. раздел `## Decomposition phase`). Decomposition — sub-phase внутри Closure (per D17 «closure-phase product-spec'а»), не отдельная пятая фаза.

После closure (включая decomposition phase либо её explicit skip/defer) — User Review Gate, потом передача в feature-spec (если оператор инициирует) или закрытие сессии.

### Session setup

При активации скилла:

1. **Glob-проверка** в настроенном расположении (по умолчанию `<project>/docs/specs/`):
   - `*product-spec.md` — существующие файлы product-spec.
   - `<product-slug>-product-backlog.md` — бэклог текущего продукта.

2. **Определить активную спеку** среди найденных. Активная = поле `status` в метаданных ∈ {draft, signed}. Замороженная = `status: shipped`.

3. **Ветвление по результату:**
   - **Активная спека `signed` без plan directory + без маркера `decomposition: skipped`** → **Entry B в decomposition phase (v0.6.0)**. Проверка: существует ли `<product-slug>-iteration-<N>-plan/` directory рядом со спекой? Если **плана нет И маркера `decomposition: skipped` нет в frontmatter спеки** → предложить:

     > *«Найдена подписанная спека [path]. Плана разбиения нет. Что делаем?*
     >
     > *1. Запустить фазу разбиения сейчас — сгенерирую план и контекстные пакеты по группам функций.*
     > *2. Открыть update-режим — править саму спеку (добавлять / удалять функции, обновлять scope cuts).*
     > *3. Закрыть сессию».*

     **Поведение по ответу:** `1` → переход сразу в decomposition phase (см. ниже), пропуская Operator dump / Probing / Шаг 12 (Closure уже было сделано). `2` → нормальный update mode (следующий branch ниже). `3` → end session, маркер не пишется.
   - **Активная спека найдена (любой другой случай — `draft`, либо `signed` с уже существующим plan directory / маркером `skipped`)** → предложить режим update: «Найдена активная спека [path], status [status]. Продолжаем работу над ней?». По умолчанию — да; оператор может переопределить («закрой active, начнём новую»).
   - **Спек нет** → первая итерация продукта, с нуля без переопределений.
   - **Только замороженные спеки** → предложить новую спеку для следующей итерации: «Найдены замороженные спеки [list]. Стартуем новую итерацию?». По умолчанию — да.
   - **Запрос на переоткрытие замороженной спеки** («открой повторно X для дополнения») → отказать: «Спеки после отгрузки заморожены и не редактируются. Если решение из iter1 нужно пересмотреть — это новое решение в новой спеке, более поздняя итерация перекрывает раньшую».

4. **Бэклог загружается в контекст** для переиспользования поддерживающих функций между итерациями и автоподсказок из parked.

5. **Подтверждение оператора** (явное или через продолжение работы) → переход в Метаданные сессии (либо сразу в decomposition phase, если выбран Entry B option 1).

### Метаданные сессии

Перед Operator dump скилл фиксирует операционные метаданные, без которых нельзя корректно создать файлы и присвоить ID. Этот шаг — операционный, не продуктовый: задаются конкретные технические вопросы с предлагаемыми default'ами, оператор подтверждает или переопределяет.

1. **Project root** — каталог, относительно которого создаются `docs/specs/`.
   - **Первая итерация продукта** (glob-проверка не нашла существующих артефактов): спросить оператора: «Где создавать spec/backlog? Default — `<cwd>/docs/specs/`. Подходит, или укажи путь?». При согласии на default — взять CWD как project root.
   - **Существующий продукт** (нашлась активная или замороженная спека): project root уже определён расположением найденных артефактов, не спрашиваем.

2. **Product-slug** — короткое kebab-case имя продукта (стабильно через все итерации, попадает в имя backlog'а и в каждое имя спеки).
   - **Первая итерация:** скилл предлагает derived slug из ответа оператора на Шаг 1.1 («Что это за продукт — одной фразой?»). Алгоритм: взять 1-3 ключевых слова, привести к kebab-case. Пример: «ERP-демо для производственной компании» → `erp-demo`; «Personal weight tracker» → `weight-tracker`. Показать оператору: «Product-slug — `erp-demo`. Подходит, или другой?». Оператор подтверждает или переопределяет.
   - **Существующий продукт:** product-slug извлекается из имени файла backlog'а / найденной спеки, не спрашиваем.

3. **Iteration-slug** — короткое имя текущей итерации (попадает в имя файла спеки).
   - Скилл предлагает derived slug из основного фокуса итерации. Если оператор обозначил конкретную тему («делаем коммуникационный модуль») → `comm-mvp`. Если фокуса нет (полноценная первая итерация широкого продукта) → `mvp`. Для последующих итераций по умолчанию — обозначение содержания (`multichannel`, `automation-v2`).
   - Показать оператору: «Iteration-slug — `comm-mvp`. Подходит?». Оператор подтверждает или переопределяет.

4. **Feature assignment** — выполняется **позже**, после Шага 4 (когда domain-функции собраны), перед Шагом 5. Здесь, на этапе метаданных, не делается. См. Feature assignment ниже после Шага 4.

После фиксации project root + product-slug + iteration-slug — переход в Operator dump.

### Operator dump

Открыть единым широким вопросом:

> *«Расскажи про продукт. Что это, для кого, какие задачи решает. Дамп всё, что в голове.»*

Если активная спека / режим update — формулировка адаптируется:

> *«Что меняется в этой итерации продукта по сравнению с уже зафиксированным? Что нужно добавить, что убрать, что переосмыслить?»*

Скилл слушает, не задаёт уточняющих вопросов. Оператор может быть кратким или подробным — обе ситуации валидны.

Дамп может быть пустым («хочу X, ничего пока не думал») — переходим к опросу без задержки.

## Probing Taxonomy

Floor checklist для brainstorm path: **6 базовых обязательных (Шаги 1-6) + финальный обязательный (Шаг 12) = 7 обязательных всего + 3 условных + 2 опциональных + 3 сквозных + 1 завершающая**.

### Universal rule

Пройти все категории, но **не обязательно** генерировать output в каждой условной/опциональной — явный N/A допустим для тех, у которых триггер не сработал. Обязательные шаги дают output всегда.

**Адаптивная глубина** — глубина шагов 1-3 варьируется по типу проекта (личный / стартап-MVP / open-source / внутренний инструмент / коммерческий). Шаги 4-12 одинаковы независимо от типа.

Каждый шаг имеет процедуру (вход → алгоритм → выход). Адаптивные уточняющие вопросы допускаются как **дополнение** к процедуре, не замена.

### Обязательные шаги

#### Шаг 1. Описание продукта

**Процедура:**
- 1.1. Спросить: «Что это за продукт — одной фразой?»
- 1.2. Уточнить тип проекта: личный / open-source / стартап-MVP / внутренний инструмент / коммерческая разработка под заказчика.
  - **Гибридные случаи** (например, демонстрационный продукт под конкретного первого заказчика; внутренний инструмент с потенциалом для коммерциализации; open-source с коммерческой версией) — оператор выбирает **доминирующий** тип и помечает гибридность в скобках. Пример записи: `тип: коммерческий (с прицелом на демо для будущих клиентов)` или `тип: внутренний инструмент (с возможной коммерциализацией позже)`. Адаптивная глубина (Шаги 1-3) следует доминирующему типу. Probe-вопросы из вторичного типа (например imagined personas из стартап-MVP в коммерческом-с-прицелом-на-демо) — задаются как **опциональные дополнения** после основного потока, по решению скилла или просьбе оператора. Без отдельного режима — фиксируется как комментарий в Section 1 спеки.
- 1.3. *Если проект не личный:* «Кто заказчик (платит) и кто пользователь (использует)? Это разные роли?»
- 1.4. *Если коммерческий или внутренний:* «Какие бизнес-цели у заказчика?» — это драйверы приоритизации, не сами функции.
- 1.5. *Если стартап-MVP или open-source:* «Какие гипотетические типы пользователей? Обоснуй.»

**Output anchor:** Section 1 (Описание продукта) спеки.

#### Шаг 2. Группы пользователей

**Процедура:**
- 2.1. Перечислить группы или роли пользователей.
  - Личный продукт: 1 группа (фиксируем явно, даже если очевидно).
  - Стартап-MVP / open-source: 2-4 гипотетические персоны.
  - Коммерческий / внутренний: обычно 2-5 реальных ролей внутри организации заказчика; минимум 1. Без искусственного раздувания списка под мифический «enterprise floor». Якорный пример contact-tracker (3 роли для команды 2-3 человек) — легитимный размер.
- 2.2. Для каждой группы — описание роли и основная цель использования продукта.
- 2.3. **Внешние actors**, не являющиеся пользователями системы (например, клиент в email-переписке, получатель push-уведомлений, автор внешнего webhook'а) — отметить отдельно, не считать группой пользователей продукта. Они влияют на buyer outcomes и контракт продукта с внешним миром (что отправляем наружу / что принимаем извне), но user journey проектируется только для внутренних групп. Записываются в отдельном sub-разделе «Внешние actors» внутри Section 2 или вообще опускаются если их роль тривиальна.

**Output anchor:** Section 2 (Группы пользователей).

#### Шаг 3. Задачи каждой группы

**Процедура:**
- 3.1. Для каждой группы — список задач, которые её пользователи хотят решать через продукт.
- 3.2. *Если групп больше одной:* пересечения задач между группами (где одна группа создаёт что-то, другая потребляет).
- 3.3. *Если коммерческая разработка:* связь задач пользователей с бизнес-целями заказчика — какие задачи продвигают какие бизнес-цели (для приоритизации).

**Output anchor:** Section 3 (Задачи пользователей).

#### Шаг 4. Перечисление пользовательских функций

Пользовательская функция = то, что юзер видит и использует, что само по себе даёт ему ценность.

**Процедура:**
1. Каждая задача из Шага 3 → кандидат в пользовательскую функцию.
2. Применить три проверки D2:
   - **Отгружаемость (independent shippability)** — отгрузить только это, будет ли value? Если ответ «нет» — не фича, переразложить.
   - **Атомарность (atomic user job)** — одна пользовательская задача на функцию? Если внутри три разных задачи — псевдо-фича (несколько функций объединены под одним ID), разбить.
   - **Переговариваемость (negotiability of inclusion)** — «делаем ли X?» осмысленный продуктовый вопрос?
3. Кандидат, не прошедший хотя бы один тест → переразложить (раздробить, объединить, отбросить).
4. **Автоподсказка из parked-бэклога.** После описания оператором каждого кандидата — скилл сканирует parked-бэклог на семантическое пересечение. **Если бэклог отсутствует или § Parked features пуст** (типично для first iteration продукта) — sub-step пропускается, переходим к Feature assignment. Иначе: совпадение → подсветить оператору: «уже было запарковано похожее F-X из iterN, level [level]. Поднять / слить / держать отдельно?». Эвристика совпадения — семантическая (LLM-суждение), не regex. Признаки совпадения: близкие названия, перекрывающиеся границы (Назначение / Входит), одинаковые upstream-зависимости.
5. Результат: список пользовательских функций с одной строкой описания на каждую.

**Output anchor:** Section 4, карточки с `Тип: пользовательская`.

#### Feature assignment (между Шагом 4 и Шагом 5)

После того как список domain-функций собран и проверен через D2-тесты — перед запуском Шага 5 скилл выполняет назначение фич. Фича — группа родственных функций (D3), не уровень иерархии. Feature-slug используется в `F-<feature-slug>-N` для всех ID функций.

**Процедура:**

1. Скилл смотрит на список собранных domain-функций и группирует по cohesion: общая user-facing область, общие пользователи, схожая семантика.
2. Для каждой группы скилл предлагает оператору feature-slug (короткий, lowercase, обычно 2-6 символов). Примеры группировки:
   - 3 функции про email и переписку с клиентом → фича `comm` (Communications).
   - 2 функции про учётки и роли (когда они появятся как enabler'ы на Шаге 5) → фича `id` (Identity).
   - 4 функции про сделки и pipeline → фича `sales` (Sales Pipeline).
   - 2 функции про contacts/companies → фича `contacts` (Contact Management).
3. Скилл показывает оператору предложенную карту «функция → фича» одной таблицей. Оператор подтверждает целиком или правит точечно (переименовать фичу, перекинуть функцию между фичами, разбить или слить фичи).
4. После подтверждения — каждая domain-функция получает финальный ID `F-<feature-slug>-N` (нумерация N — порядок внутри фичи). Enabler'ы из Шага 5 будут получать ID по тем же фичам (часто появляется новая фича `id` для общих identity/auth-enabler'ов).

**Эвристика классификации фич по категориям Mermaid-визуализации.**

После назначения фич и сбора enabler'ов на Шаге 5 — фичи относятся к одной из трёх категорий, которые используются как subgraph группы в Mermaid Section 4:

- **Domain** — фича реализует core value proposition этого продукта. То, ради чего пользователь приходит. То, что описано в Section 1 (Описание продукта) и Section 3 (Задачи пользователей) как user-facing задачи.
- **Infra (load-bearing enablers)** — фича load-bearing для domain-фич: domain-функции **используют её поведение** (auth check, permission lookup, tenant resolution) для своей работы. Сам value пользователю не даёт, но без неё domain не работает.
- **Passive (cross-cutting)** — фича реализует сквозную политику поверх остальных (audit log применяется ко всем mutating actions; notifications hub слушает события всех фич). Cross-cutting concern, не вызывается явно из domain-функций, а активируется на их события.

Классификация — **LLM-суждение** скилла на основе Section 1-3 + содержания карточек функций. **При неоднозначности** (фича могла бы быть Infra или Domain — типичный hybrid-кейс) скилл предлагает классификацию оператору одной репликой: «*Фичу Identity в этом продукте классифицирую как [Infra / Domain] потому что [reasoning]. Подтверждаешь или переклассифицировать?*». Оператор подтверждает или переопределяет.

**Категория фичи — product-relative, не absolute.** Одна и та же фича (Identity, Permissions) в разных продуктах может попасть в разные категории. В end-product (CRM, ERP, личный трекер) Identity — это Infra. В infrastructure-spec (платформа auth для нескольких продуктов; permissions-as-a-service) та же Identity сама становится Domain — она и есть продукт. Категория определяется ролью **в этом продукте**, не absolute классификацией.

**Hybrid product (фича с двумя ролями одновременно).** Существуют продукты, где одна и та же фича играет обе роли: CRM с public auth API; SaaS, у которого identity-слой также продаётся отдельно; ERP с public webhooks. Правило: **одна фича = одна категория** (визуализация остаётся однозначной). Выбирается **dominant роль** — та, которая отражает основное value proposition продукта. Secondary роль фиксируется комментарием под Mermaid'ом в Section 4 (пример: «*Identity classified as Infra (dominant — internal use). Также exposes public auth API as secondary value proposition (см. F-id-3)*»). При неоднозначности dominant'а — скилл эскалирует оператору (продуктовое решение, P6).

**Pivot guidance при перестройке feature map.**

Если в ходе Шагов 5-6 или premortem'а в Шаге 11 происходит существенный pivot feature map (новая фича появилась, функция перенесена между фичами, фичи слиты или разделены) — **до перехода к Шагу 12** скилл повторяет feature assignment процедуру для затронутых функций, пересматривает категориальную классификацию и пересчитывает F-ID функций по обновлённому feature-slug'у. Без этого DAG зависимостей и Mermaid становятся несогласованными со scope'ом. Это детерминированное действие (P6) — не требует продуктового вопроса оператору, кроме случаев когда новые фичи создают неоднозначность классификации.

**Обновление cross-references при F-ID renumber.** Одновременно с пересчётом F-ID скилл обновляет все cross-references к старому F-ID в спеке (поля `Зависит от` и `Не входит` карточек, упоминания в Sf-развилках, CC-политиках и Section 9 сценариях) одним find-replace pass'ом. Если оператор не уверен в результате — скилл показывает before/after preview по затронутым строкам перед применением.

**Note:** для очень малых продуктов (1-2 функции) фича может быть одна-единственная. Это нормально, не пытаться искусственно дробить.

#### Шаг 5. Вывод поддерживающих функций (сверху вниз)

Поддерживающая функция = то, что нужно, чтобы пользовательские функции работали (учётная запись, права, восстановление пароля, нотификации, поиск, журнал действий, экспорт). У них тоже есть продуктовые решения, поэтому они в продуктовой спеке.

**Процедура:**

**0. Operator-facing объяснение процесса + visibility пропущенных probes.**

Перед запуском probes скилл говорит оператору две вещи (точные формулировки — гайды, агент адаптирует под контекст продукта).

*Часть 1 — объяснение процесса:*

> «Сейчас по каждой пользовательской функции пройдусь по списку продуктовых вопросов про инфраструктуру (учётки, права, нотификации, поиск, журнал действий, восстановление пароля, изоляция нескольких клиентов на одной системе, экспорт). По каждому вопросу три сценария: если ответ очевиден из контекста — приму дефолт и явно покажу его как «дефолт: X, альтернатива: Y, подтверждаешь?»; если ответ неочевиден или есть несколько разумных вариантов — спрошу тебя; если вопрос не применим к этому продукту по моему контексту — пропущу, но скажу об этом сейчас (см. ниже).»

*Часть 2 — visibility пропущенных вопросов (закрывает silent skip):*

> «Из 8 областей вопросов в этом продукте по моему контексту не применимы: <список с причиной за каждую>. Например: изоляция нескольких клиентов — продукт для одного клиента-демо; журнал действий — нет требований по соответствию нормативам; экспорт — нет внешних потребителей данных. Если что-то из этого ты хочешь обсудить явно — скажи сейчас, до запуска вопросов.»

Probes, помеченные «не применимы» здесь — не появляются в выводе Шага 5. Probes, по которым оператор поднял возражение — добавляются обратно в список и обрабатываются как обычные probes.

**1. Probe-таблица + selectivity + visible defaults.**

Для каждой пользовательской функции — пройти список вопросов в продуктовом языке (probe table per D9):

| Инженерный вопрос (НЕ задавать) | Продуктовый эквивалент (задавать оператору) | Поддерживающая функция, которая выводится |
|---|---|---|
| Нужен ли auth? | Один тип юзера или разные роли? | Учётные записи |
| Какой auth method? | Самостоятельная регистрация или invite-only? | Регистрация |
| RBAC или ABAC? | Все админы равны или есть градации прав? | Модель прав |
| Multi-tenancy? | Несколько изолированных компаний делят инстанс? | Изоляция клиентов |
| Какой password reset? | Юзер может забыть пароль и восстановить сам? | Восстановление пароля |
| Нотификации? | Юзер хочет узнавать о событиях, когда он не в системе? | Каналы нотификаций |
| Search infra? | Юзеру нужно найти запись среди многих? | Поиск |
| Audit log? | Нужно знать, кто что когда сделал? | Журнал действий |

**Selectivity probe-таблицы.** Не все 8 probes из таблицы D9 нужно задавать каждой domain-функции. Скилл выбирает релевантные по контексту самой функции и типа проекта. Например: multi-tenancy probe не задаётся для single-customer демо или личного продукта; password reset probe не задаётся для одной session-based функции без auth; audit log probe — только если домен высокоставочный (финансы / медицина / B2B-операции с историей действий) либо оператор явно поднял тему. Probes пропущенные по этой эвристике уже зафиксированы в Sub-step 0 Часть 2 — повторно не упоминаются.

**Visible defaults (правило обязательной visibility).** Когда скилл presumes ответ на probe — relevant probe, но дефолт очевиден из контекста — он **обязан** показать дефолт оператору как `(дефолт: X; альтернатива: Y; подтверждаешь?)` в момент применения дефолта. Молчаливый дефолт запрещён. Этот гейт — на уровне отдельных probes; собранная таблица всех решений финализируется в Sub-step 6 ниже.

**2.** Каждое продуктовое «да» порождает кандидата на поддерживающую функцию.

**3. Проверка committed-бэклога (переиспользование поддерживающих функций между итерациями).** **Если бэклог отсутствует или § Committed features пуст** (first iteration продукта) — sub-step пропускается, переходим к sub-step 4. Иначе: перед созданием новой enabler-карточки скилл проверяет committed-бэклог на совпадение по указателю (Входит / Не входит). Дерево решений:
- **Совпадение (Входит покрывает нужное)** → предложить **переиспользование**: «У тебя F-id-2 (Права доступа) уже в committed-бэклоге. Покрывает плоские роли admin/user, что нужно. Используем без новой карточки?». Оператор подтверждает → новая domain-карточка получает `Зависит от: F-id-2`, новый enabler не создаётся.
- **Частичное совпадение (Входит покрывает основное, но нужно добавить)** → предложить **расширение**: «F-id-2 покрывает плоские роли, но не иерархические (роли наследуются друг от друга). Создаём F-id-2-ext в этой итерации?». Оператор подтверждает → новая карточка F-id-2-ext, зависит от F-id-2.
- **Несовпадение (Не входит явно содержит нужное)** → предложить **новую карточку**: «F-id-2 не покрывает атрибутные роли (права от атрибутов записи, не от роли пользователя). Создаём F-id-3 (атрибутные права)?».
- **Неоднозначно (указателя недостаточно)** → скилл **читает исходную спеку по требованию** для полной карточки enabler'а, делает финальное решение.

**4. Автоподсказка из parked-бэклога** (если committed-проверка не дала совпадения). **Если бэклог отсутствует или § Parked features пуст** (first iteration) — sub-step пропускается, переходим к sub-step 5. Иначе: скилл сканирует parked-бэклог на семантическое совпадение — совпадение → подсветить оператору, как в Шаге 4.

**5.** Промежуточный список поддерживающих функций с зависимостями обратно к пользовательским, которым они служат. Финализируется в Sub-step 6.

**6. Structured review block в конце Шага 5.**

После прохождения probes по всем domain-функциям и формирования полного списка enabler'ов — скилл показывает оператору structured review block:

```
Шаг 5 завершён. Выведено N поддерживающих функций. По каждой — ключевые решения:

| ID       | Название           | Ключевые решения                              | Источник              | Действие                              |
|----------|--------------------|-----------------------------------------------|-----------------------|---------------------------------------|
| F-id-1   | Учётные записи     | самостоятельная регистрация; magic-link reset | дефолт                | подтвердить / обсудить / альтернатива |
| F-id-2   | Права доступа      | плоские роли admin/user/viewer (admin + 2)    | обсуждение в Шаге 5   | подтверждено / переоткрыть            |
| F-notif-1| Нотификации        | email-канал; нет in-app push                  | дефолт                | подтвердить / обсудить / альтернатива |
| ...      | ...                | ...                                           | ...                   | ...                                   |

Пропущены целиком (повтор из начала Шага 5):
- изоляция нескольких клиентов — продукт для одного клиента-демо
- журнал действий — нейтральный домен без требований по нормативам
```

Колонки:
- **Источник** — `дефолт` (presumed default) или `обсуждение в Шаге 5` (probe задана как вопрос, оператор ответил).
- **Действие** — для строки `Источник = дефолт`: «подтвердить / обсудить / альтернатива». Для строки `Источник = обсуждение`: «подтверждено / переоткрыть».
- **Пропущенные probes** — повтор из Sub-step 0, чтобы оператор видел их и при review.

Свободная форма «возражаешь?» запрещена. Только structured presentation. Если оператор подтверждает все одним «подтверждаю всё» — валидно, но скилл сначала показывает таблицу.

**Recovery loop при «обсудить» / «альтернатива» / «переоткрыть».** Скилл возвращается к probe этого enabler'а с уточняющим вопросом. После резолва — обновляет строку таблицы и продолжает по остальным.

**Если оператор отметил ≥2 строк к обсуждению за раз** — скилл обрабатывает последовательно (по одной), показывая обновлённую таблицу после каждого резолва. Параллельное обсуждение нескольких enabler'ов в одном круге не поддерживается: пересечения вопросов между ними могут породить нежелательный cascade или путаницу контекста.

**Если в ходе recovery появляется новый enabler** (например, оператор отвечает «нет, нужна не плоская модель, а иерархическая с inheritance» → новый F-id-3):

1. К новому enabler'у применяются **тесты D2** (атомарность / отгружаемость / переговариваемость) — как к обычной enabler-функции.
2. К новому enabler'у применяется **полный probe-блок Шага 5** с теми же visible defaults / гейтами.
3. **Feature assignment** — новый enabler получает feature-slug (обычно из существующих фич; если требуется новая фича — возврат к Feature assignment после Шага 4 с пересмотром списка).
4. Новый enabler добавляется в таблицу sub-step 6 как новая строка, **источник = `обсуждение в Шаге 5`**.

**Maximum cascade depth = 2.** Если probe нового enabler'а порождает ещё один новый enabler — оба добавляются. Третий уровень cascade (новый enabler из probe-обсуждения нового enabler'а) **запрещён в одном Шаге 5**. Если оператор хочет добавить третий уровень — это сигнал что scope взрывается; скилл говорит «достигнут предел вглубь Шага 5, новые enabler'ы требуют отдельного прохода. Останавливаемся, фиксируем текущий список, в Шаге 12.2 проверка целостности подсветит если что-то требует return в Шаги 4-5».

**Output anchor:** Section 4, карточки с `Тип: поддерживающая`.

#### Шаг 6. Границы каждой функции

Для каждой функции (пользовательской и поддерживающей) заполнить три поля:

- **Входит:** что включаем в эту функцию в этой версии.
- **Не входит (но это отдельная функция):** что вынесено в другую функцию роадмапа.
- **Не делаем вообще:** что отрезано на уровне продукта.

**Процедура:**
1. Уточняющие вопросы оператору: «Какой минимально полезный кусок этой функции? Где её разумная граница? Что точно не делаем в этом продукте?»
2. **Отметка о совпадении при парковке.** При обсуждении «выносим в parking» — скилл сканирует parked-бэклог на семантическое пересечение. Совпадение → подсветить: «уже было запарковано похожее F-X из iterN, слить или держать отдельно?».
3. Эвристика для оператора: если в product-spec нужно расписать функцию больше 5-8 строк, чтобы решить «включаем в MVP» — детализация ушла глубоко. Подняться обратно до scope cut. Дальнейшее — feature-spec.

**Output anchor:** Section 4 карточки, поля Входит / Не входит / Не делаем вообще.

### Условные шаги (запускаются при срабатывании триггера)

#### Шаг 7. Сквозные политики — inheritance-aware

**Триггер:** ≥2 in-scope функций в этой итерации **ИЛИ** новая функция вводит сквозную политику (audit log, нотификации, поиск, экспорт, файловое хранилище и т.п.), которой не было в предыдущих итерациях.

**Процедура:**
- 7.1. Скилл читает бэклог § Standing policies + спеки предыдущих итераций — список применимых уже-известных политик.
- 7.2. Для каждой in-scope функции этой итерации скилл проверяет: попадает ли под существующие standing-политики? Если да — функция **наследует политику** (в карточке добавляется поле `Inherits: CC-X from <previous spec>`).
- 7.3. Если в текущей итерации появляются новые сквозные политики, **не покрытые** существующими — задать вопрос оператору («Единый audit log поверх всех функций или у каждой свой?», «Общая система нотификаций или каждая функция шлёт свои?»). Output: новая CC-политика в spec § Сквозные политики, потенциально новая поддерживающая функция (общий сервис журнала / общий узел нотификаций).

**Если все сквозные политики этой итерации покрыты существующими standing-политиками — Шаг даёт результат «только унаследовано, новых политик в спеке нет».** Это валидное завершение, не пропуск.

**Output anchor:** Section 6 (Сквозные политики).

#### Шаг 8. Жизненный цикл данных — inheritance-aware

**Триггер:** in-scope функция этой итерации хранит постоянные данные **И** либо нет существующих политик жизненного цикла (первая такая функция в продукте), либо новая функция вводит аспект жизненного цикла, не покрытый существующими.

**Процедура:**
- 8.1. Скилл читает существующие политики жизненного цикла (если есть в бэклоге или предыдущих спеках).
- 8.2. Для каждой in-scope функции, которая хранит постоянные данные — проверка: существующие политики хранения / GDPR / миграции покрывают?
- 8.3. Если новая функция вводит новый аспект жизненного цикла данных — задать вопрос («Что происходит с данными пользователя, когда он уходит?», «Как долго хранить логи?», «Нужны ли право на забвение и экспорт?», «Удалять данные совсем или просто помечать как удалённые?»). Output: новая политика жизненного цикла в спеке этой итерации, возможно новая поддерживающая функция (сервис retention, экспорт GDPR), стратегические развилки.

**Если все аспекты жизненного цикла покрыты существующими политиками — Шаг даёт результат «только унаследовано».** Аналогично Шагу 7.

**Output anchor:** Section 6 (Сквозные политики), часть про lifecycle.

#### Шаг 9. Взаимодействия между группами пользователей

**Триггер:** групп больше одной (Шаг 2 даёт ≥2).

**Процедура:** задать оператору: «Какие сценарии требуют участия двух и более групп? Кто видит данные кого? Какие функции доступны только определённым группам?»

**Output anchor:** Поле `Доступно группам` в карточках Section 4; развилки в межгрупповых сценариях — Section 5.

### Опциональные шаги (только при явном основании)

#### Шаг 10. Соответствие нормативным требованиям

**Триггер:** домен подлежит регулированию (медицина, финансы, образование, обработка персональных данных GDPR, доступность WCAG).

**Процедура:** задать оператору: «Какие требования законов или стандартов применимы? Кого они затрагивают? Какие функции продукта обязаны им соответствовать?»

**Не выдумывать требования для нейтральных доменов.** Если триггер не сработал — явный N/A.

**Output anchor:** Жёсткие политики в Section 6.

#### Шаг 11. Базовые требования к нагрузке и доступности

**Триггер:** оператор поднимает явно либо домен высокоставочный.

**Процедура:** задать оператору: «Порядок числа пользователей? Ожидания по скорости отклика? Гарантии доступности? Объём данных? Работа без интернета или только онлайн?»

В продуктовую спеку попадает только то, что заказчик / оператор осознанно выбирает как продуктовое решение (например: «обязательно работает офлайн»). Конкретные числа нагрузки — материал арх-спеки.

**Output anchor:** Section 8 (Базовые требования к нагрузке и доступности).

### Финальный обязательный шаг

#### Шаг 12. Подтверждение scope + проверка целостности + миграция в бэклог

Запускается **после Шагов 4-11 + завершающей проверки (премортема)**. Если премортем породил новые функции / forks / scope cuts — они должны быть добавлены через Шаги 4-6 / Section 5 / scope cut обновление **до** запуска Шага 12. Шаг 12 закрывает миграцию и переход статуса; функции, добавленные после него, в текущую спеку не попадут.

**Процедура:**

- **12.1. Подтверждение scope.** Скилл показывает оператору финальный сводный список: in-scope функции (с карточками), parked-в-этой-итерации (с level), deferred-развилки, rejected. Оператор подтверждает или правит.

- **12.2. Проверка целостности** (скилл выполняет автоматически):
  - **Проверка-сирота** — enabler в in-scope, ни одна domain-функция от него не зависит → подсветить: «F-X в scope, никто не использует — точно нужен?».
  - **Разорванная замкнутость** — domain-функция в in-scope, её enabler в parked / out-of-scope → подсветить: «F-Y зависит от F-Z, который parked — резолви».
  - **Циклические зависимости** — цикл в графе зависимостей → блокирующая ошибка с требованием резолва.

  **Recovery-loop при failed integrity check.** Если любая из проверок выявила проблему — спросить оператора, что делать:
  - **Сирота** — убрать enabler в parked / out-of-scope (return в Шаг 6 для обновления Status), либо подтвердить что он нужен по другой причине (тогда просто принять предупреждение как noted).
  - **Разорванная замкнутость** — три варианта:
    - Добавить недостающую функцию в in-scope (return в Шаг 4 или 5 для создания карточки) → пересобрать DAG → повторить 12.2.
    - Убрать broken функцию из in-scope (return в Шаг 6 для перевода в parked) → пересобрать DAG → повторить 12.2.
    - Переделать поле `Зависит от`, если зависимость была ошибочной.
  - **Цикл** — резолвить через одного из участников цикла: убрать одну из зависимостей, объединить функции в одну, или вынести общую часть в третью функцию-предка. После резолва — повторить 12.2.

  Recovery-loop возвращается на Шаг 12 столько раз, сколько нужно, пока integrity check не пройдёт чисто.

- **12.3. Миграция в бэклог при закрытии:**
  - in-scope → бэклог § Committed (асимметричные указатели по типу — см. Backlog mechanics ниже).
  - parked-в-этой-итерации → бэклог § Parked (полное содержимое + поле level).
  - deferred-развилки → бэклог § Deferred (полное содержимое).
  - rejected → бэклог § Out-of-scope (one-liner).
  - resolved-развилки остаются в spec § Стратегические развилки.
  - Iteration index в бэклоге обновляется новой записью.

  **Note: terminology «Committed» vs «Shipped».** В backlog'е секция называется `§ Committed` (а не `§ Shipped`), потому что миграция в неё происходит при переходе спеки в `signed` — это commitment по продуктовому дизайну, не реальная отгрузка в production. Указатели получают тег `[committed iter<N>: <slug>]`. Статус спеки `shipped` (после реальной отгрузки) — отдельная семантика; в Iteration index он отражается полем status, не названием секции. Это сохраняет когнитивную ясность: signed = committed (дизайн закрыт), shipped = реально отгружено в production.

- **12.4. Переход статуса.** При подтверждении оператора — статус спеки автоматически переходит `draft` → `signed` (готов к feature-spec). Переход `signed → shipped` — позже, по явной команде оператора («iter X отгружена / iter X в production»). Скилл применяет переходы детерминированно, без эскалации (см. Operational Rules).

- **12.5. Entry A gate в decomposition phase (v0.6.0).** Сразу после flip'а `draft → signed` скилл задаёт оператору **explicit gate** на запуск decomposition phase:

  > *«Спека подписана. Готов перейти к фазе разбиения на блоки — сгенерирую план реализации и контекстные пакеты для feature-spec сессий по каждой группе функций. Запустить сейчас?*
  >
  > *1. Да — генерирую план и пакеты, потом сессия закрыта.*
  > *2. Отложу — план не делаем сейчас, потом сам запущу (новой сессией на эту же спеку).*
  > *3. Не нужно — план не требуется для этой спеки (например, single-feature продукт, разбивать нечего)».*

  **Поведение по ответу:**
  - `1` (да) → переход в decomposition phase (см. раздел `## Decomposition phase` ниже).
  - `2` (отложу) → end session, маркер `decomposition` не пишется. Entry B при следующей сессии естественно detect'нёт «signed + no plan» и предложит снова.
  - `3` (не нужно) → write маркер `decomposition: skipped` в frontmatter спеки. Entry B видит маркер и не auto-prompt'ит. Operator может override: удалить маркер вручную из метаданных, либо сказать «запусти decomposition phase» в новой сессии (skipping detection).

  Silent skip запрещён — это нарушение urok'а v0.5.0 (visibility пропущенных решений). Gate явный, оператор всегда видит выбор.

**Output:** spec готова к User Review Gate. Если operator выбрал option 1 — далее идёт decomposition phase, и User Review Gate happens после её завершения (плюс plan directory). Бэклог обновлён, frozen invariant сохраняется.

### Сквозные проверки (постоянно активны во время всей сессии)

#### CCP1. Выявление неявных критериев

Если в ответе оператора всплывает расплывчатое слово (быстро, часто, много, опасный, похоже, естественно) — задать уточняющий вопрос: переводим в число, в правило-чеклист, или принимаем как принципиально нечёткое и закрепляем якорным примером.

#### CCP2. Проверка на противоречия

После каждого нового ответа сверять с уже записанной структурой. Если новый ответ противоречит чему-то — поднять конфликт оператору, не записывать молча:

> *«В Шаге 4 ты сказал X, в текущем ответе предполагается Y. Конфликт. Какое решение верно / нужно ли пересмотр?»*

#### CCP3. Классификация стратегических развилок

Когда в ответах всплывает развилка-кандидат — применять **тест D5**:

1. Меняет ли feature list? Ветки развилки приводят к разным спискам функций в роадмапе?
2. Меняет ли cross-feature deps? Ветки реорганизуют граф зависимостей (новые связи, новые корни)?

Любое «да» → стратегическая развилка → секция 5 спеки. Если оба «нет» — feature-уровень, не записывается в продуктовую спеку, пойдёт в feature-spec.

В серых зонах оператор решает явно («это будет product-spec fork» / «это уйдёт в feature-spec»). Скилл подсвечивает серую зону, не выбирает за оператора.

**Автопроверка по deferred-развилкам.** Также проверка пересечения кандидата-развилки с deferred-развилками из бэклога. Совпадение → подсветить оператору: «связано с deferred Sf-Y из iterN, резолвить сейчас или продолжать как отдельную развилку?».

Перед завершением сессии — отдельный финальный проход по всем накопленным развилкам для подтверждения корректной классификации.

### Завершающая проверка — премортем на 6 месяцев

Запускается **после Шагов 4-11, до Шага 12.**

В конце основного потока задать вопрос:

> «Представь — продукт запущен, прошло 6 месяцев. Что пошло не так? Сценарии: пользователи не пришли / пришли но ушли / конкурент съел рынок / технический долг сломал скорость разработки / сам оператор отказался поддерживать продукт.»

Каждое представимое осложнение → потенциально новая функция, стратегическая развилка или граница, которую упустили.

**Обработка дельты от премортема.** Если премортем порождает:
- **Новую функцию** (domain или enabler) → return в Шаг 4 или Шаг 5 для добавления карточки. После — return сюда для верификации, что новых дельт нет.
- **Новую стратегическую развилку** → добавить в Section 5 через CCP3 (тест D5).
- **Новый scope cut** (что-то вынести в parked / out-of-scope) → return в Шаг 6 для обновления карточек Входит / Не входит / Не делаем вообще.
- **Явное решение «откладываем в parked / deferred»** — записать соответствующую парковку, не добавлять в in-scope.
- **Дельты нет.** Премортем-сценарии есть, но все mitigations external (контракты с заказчиком, маркетинг, организационные меры) или non-product (инфраструктура, операционные процессы). Это **валидный исход** — премортем закрепляется в Section 11 как сценарии-предупреждения, без порождения функций / развилок / scope cuts. Прямой переход к Шагу 12.

После того как все порождённые премортемом дельты обработаны (или явно отклонены оператором, или дельты не было) — запуск Шага 12.

**Output anchor:** Section 11 (Что может пойти не так).

### Условие завершения сессии

Перейти к закрытию можно только когда (порядок шагов явный):

1. Все обязательные шаги 1-6 дали результат.
2. Feature assignment выполнен.
3. Все условные шаги 7-11 с сработавшими триггерами дали результат либо явно помечены «не применимо».
4. CCP3 прошла финальный проход.
5. CCP2 — нет открытых конфликтов.
6. Премортем проведён, порождённые им дельты обработаны (через return в Шаги 4-6 / Section 5 / scope cuts) или явно отклонены.
7. Шаг 12 (после премортема) — подтверждение scope + проверка целостности + миграция в бэклог.

## Backlog mechanics

Бэклог хранит реестр функций, живущий через все итерации продукта, с **асимметричной дисциплиной размера записей** (защита от неограниченного роста по уроку из архитектурного документа проекта).

### Содержание бэклога по типам записи

**Iteration index** — таблица с одной строкой на итерацию:

| Дата | Slug | Статус | Spec |
|------|------|--------|------|
| 2026-05-10 | erp-mvp | shipped (2026-07-15) | docs/specs/2026-05-10-erp-mvp-product-spec.md |

**Указатель на committed-enabler** (5-6 строк):

```markdown
### F-id-2. Permissions [committed iter1: erp-mvp]
**Фича:** Identity
**Type:** enabler
**Назначение:** базовая модель прав доступа на уровне ресурсов
**Входит:** flat-роли (admin/user/viewer); per-resource permissions; assign/revoke/list
**Не входит:** иерархические роли с inheritance; ABAC; audit log
**Spec:** docs/specs/2026-05-10-erp-mvp-product-spec.md § F-id-2
```

**Указатель на committed-domain** (3 строки):

```markdown
### F-comm-1. Email-conv [committed iter1: erp-mvp]
**Фича:** Communications
**Type:** domain
**Назначение:** двусторонние email-разговоры оператора с клиентами с threading
**Spec:** docs/specs/2026-05-10-erp-mvp-product-spec.md § F-comm-1
```

**Parked-функция** (полное содержимое + поле level):

```markdown
### F-comm-X. Auto-classification (level: detailed)
**Status:** parked
**Originating analysis:** docs/specs/2026-08-15-erp-multichannel-product-spec.md (Секция 4)
**Reason for parking:** ML-стоимость не вписалась в бюджет iter2
**Carry-over candidate for:** iter3+
[full card content]
```

Уровни parked: `detailed / partial / note / fragment`. Оператор выбирает level при парковке (скилл спросит: «насколько детально хочешь сохранить?»).

**Deferred-развилка** — полная карточка как было в спеке + причина откладывания + ссылка на исходную спеку.

**Out-of-scope (rejected)** — one-liner: `F-Z. Mobile app native — отвергнуто iter1, причина: scope MVP не выдерживает; альтернатива — responsive web.`

**Standing-политики** — указатель (1 строка + ссылка на исходную спеку).

### Асимметричная дисциплина указателей — обоснование

| Тип записи | Формат | Размер |
|---|---|---|
| Указатель на committed-enabler | ID + Name + Фича + Type + Назначение + Входит + Не входит + spec ref | 5-6 строк |
| Указатель на committed-domain | ID + Name + Фича + Type + Назначение + spec ref | 3 строки |
| Parked-функция (любой Type) | Полный контент + level + причина | varies |
| Deferred-развилка | Полная карточка + причина откладывания | varies |
| Out-of-scope rejected | One-liner + причина | 1 строка |
| Запись в iteration index | Дата + slug + статус + spec ref | 1 строка |
| Указатель на standing-политику | ID + Name + spec ref | 1 строка |

**Почему асимметрия.** Shipped растёт линейно во времени без верхней границы (каждая итерация добавляет, ничего не убывает) → должны быть указатели. Parked / deferred — ограничены по смыслу (поглощаются в новых итерациях по мере резолва) → могут быть полным содержимым. Domain shipped vs enabler shipped: переиспользование enabler'ов требует совпадения scope (нужны Входит / Не входит), переиспользование domain'ов чаще решается на уровне названия → разные форматы указателей.

Триггер автоматического перевода в standing: в спеке iter2 вводится та же политика (по смыслу), что была в iter1 — скилл предложит «эта политика переезжает в standing, дублировать в каждой спеке не имеет смысла».

### Переиспользование поддерживающих функций между итерациями (Шаг 5 detail)

Расширенное дерево решений для каждой новой domain-функции на Шаге 5 — см. Шаг 5 § 3 выше.

Указатель на committed-domain (минимальный) обычно используется только для дедупликации по названию в Шаге 4. При запросе по scope — чтение спеки по требованию (редкий кейс).

### Автоподсказка из parked-бэклога

Скилл сканирует parked-бэклог на семантическое пересечение в трёх местах:
- **Шаг 4** — после описания оператором кандидата в domain-функцию.
- **Шаг 5** — после кандидата enabler, если committed-проверка не дала совпадения.
- **Шаг 6** — при обсуждении «выносим в parking».

Эвристика совпадения — семантическая (LLM-суждение), не regex. Признаки совпадения: близкие названия, перекрывающиеся границы, одинаковые upstream-зависимости.

Ложные срабатывания / пропуски — оператор корректирует устной фразой («нет, другая фича»), не блокирует процесс.

Аналогично для **deferred-развилок** в CCP3 — совпадение → подсветить «связано с deferred Sf-Y из iterN, резолвить сейчас или продолжать как отдельную развилку?».

### Режим update (D14 mechanics)

Если на Session setup определена активная спека → режим update применяется к текущей спеке и бэклогу.

**Поддерживаемые операции:**

- **Добавить функцию** (новый F-X): probe Шаги 4-6 для новой; Шаг 5 enabler closure; обновление спеки.
- **Удалить функцию**: проверка downstream-зависимостей (никто не зависит?); если зависят — резолвить.
- **Разделить функцию** (X → A + B): добавить A + добавить B; удалить X; перенаправить зависимости; пересмотр целостности.
- **Слить функции** (A + B → C): добавить merged; удалить обе; перенаправить зависимости.
- **Обновить scope cuts**: редактировать карточки (поля Входит / Не входит), без каскада.
- **Добавить / отредактировать CC-политику**: новая или обновлённая запись в Section 6.
- **Резолвить OPEN-развилку**: применить изменения по выбранной ветке (add/remove функций); обновление списка функций.

**Mechanics:**
- Read tool загружает спеку + бэклог в контекст.
- LLM понимает структуру семантически, редактирует точечно через Edit tool.
- Проверка целостности после каждой операции (сирота / разорванная замкнутость / циклы).
- Mermaid в Section 4 регенерируется LLM'ом при изменении зависимостей.
- Представление diff'а — через «было/стало» в предложении оператору перед применением.
- Гейт подтверждения — на логическую операцию (split = одно подтверждение, не три отдельных).

**Без structured parsing.** Скилл работает с markdown как с текстом. Mermaid helper script alternative припаркован в OQ14 (если LLM-генерация Mermaid окажется ненадёжной на больших графах — пересмотрим).

**Frozen invariant** — shipped-спеки не редактируются (D18). Если оператор пытается «открыть замороженную спеку для дополнения» — отказать: «Спеки после отгрузки заморожены и не редактируются. Если решение из iter1 нужно пересмотреть — это новое решение в новой спеке, более поздняя итерация перекрывает раньшую».

### Поле status в спеке (D18)

Метаданные спеки содержат поле `status` с тремя значениями:

- `draft` — спека в дизайне (оператор работает в текущей сессии).
- `signed` — дизайн закрыт, перешли к feature-spec / impl. Активная спека, можно редактировать.
- `shipped` — итерация отгружена, спека **заморожена**, не редактируется.

**Переходы статуса** (детерминированные действия скилла, без эскалации оператору):

- `draft → signed` — автоматически по завершении Шага 12 + User Review Gate.
- `signed → shipped` — по явной команде оператора «iter X отгружена / iter X в production». Скилл применяет детерминированно.
- `shipped → anything` — запрещено. Скилл блокирует попытки редактирования.
- `signed → draft` — оператор может вручную через правку метаданных, если решает переоткрыть дизайн (скилл не борется, но и не инициирует).

Активная = `draft` или `signed`. Замороженная = `shipped`.

**Связь с дисциплиной P6 ARCHITECTURE.md.** Переходы статуса, проверки целостности, валидация формата и миграция в бэклог — детерминированные действия. Скилл выполняет их без вопросов оператору. Эскалация — только для продуктовых решений (что включаем, как режем scope, как резолвим конфликт).

## Decomposition phase

Closure sub-phase (v0.6.0), запускается после Шага 12 по решению оператора. Производит план реализации (`roadmap.md`) + контекстные пакеты для feature-spec сессий (`bundle-<feature-slug>.md` × N) внутри subdirectory `<product-slug>-iteration-<N>-plan/` рядом со спекой.

**Зачем нужна:** product-spec даёт спеку на N функций с DAG зависимостей через поле `Зависит от`. feature-spec ждёт на вход одну фичу с локальным контекстом. Decomposition phase — мост: уточняет границы фич (после initial Feature assignment Шага 4-5), топологически сортирует по feature-level deps, генерирует self-contained context bundles per фичу.

**Scope в v0.6.0:** только **initial decomposition** в Fresh + Update session modes. Re-decomposition при изменениях active спеки и new iteration mode handling — defer в v0.6.3+.

### Trigger logic

Два entry points (детально описаны в Шаге 12.5 и Session setup соответственно):

- **Entry A — explicit gate в Шаге 12.5** (после fresh mode closure). Сразу после flip'а `draft → signed` оператор выбирает: запустить сейчас / отложить (без маркера) / не нужно (маркер `decomposition: skipped`).
- **Entry B — session setup detection** (update mode initial decomposition). При обнаружении `signed` спеки без plan directory и без маркера — оператор выбирает: запустить decomposition / открыть update mode / закрыть.

При triggering — переход в sub-phases ниже по порядку: feature boundary refinement dialog → topological sort (+ cycle handling если нужно) → bundle generation → standing-policy candidates → output artefacts (write files).

### Sub-phase 1: Feature boundary refinement dialog

Refines initial feature assignment (Шаги 4-5) через splits/merges при необходимости. По умолчанию — фичи остаются как назначены, без изменений.

**Sub-step 1.1 — Build default feature map.** Default — feature assignment из Шага 4-5 без изменений: каждая фича из спеки → один блок реализации с тем же slug'ом. Все функции фичи входят в её блок.

**Sub-step 1.2 — Detect split candidates** (LLM-suggestion per фича). Триггер: фича с ≥5 функциями И функции внутри образуют ≥2 качественно разные подгруппы по семантике (lifecycle phases типа inbound/outbound; templates vs execution; разные user groups served). Detection — на основе текста карточек (Назначение + Входит + Доступно группам). Не regex, не keyword match.

**Sub-step 1.3 — Detect merge candidates** (deterministic + LLM). Триггеры: межфичевая dependency density ≥3 deps между парой фич (deterministic count из `Зависит от`), ИЛИ оператор в Operator dump'е / Probing'е явно упомянул cross-feature lifecycle (LLM-detection из session conversation).

**Sub-step 1.4 — Presentation (единая таблица).** Скилл показывает default map + flagged candidates:

```
Карта фич (по умолчанию — фичи из Feature assignment Шага 4-5 без изменений):

| # | Фича | Функции | Кандидат к ревью |
|---|------|---------|-------------------|
| 1 | id | F-id-1..4 (4) | — |
| 2 | org | F-org-1..2 (2) | — |
| 3 | tickets | F-tickets-1..5 (5) | разбить? lifecycle: подача (1+2) vs исполнение (3+4+5) |
| 4 | soft | F-soft-1..5 (5) | разбить? шаблоны (1+4+5) vs выполнение (2+3) |
| ... | | | |

Межфичевые связи:
- tickets ↔ comm: 4 зависимости между F-tickets-* и F-comm-2. 
  Объединить как одну фичу жизненного цикла «заявка + переписка»?

По каждой строке твоё решение: подтвердить / разбить / слить с другой / переименовать / отменить разбиение.
```

**Sub-step 1.5 — Recovery loop на действия оператора:**

- **Разбить** — операнд: исходная фича + распределение функций по новым подфичам. Скилл предлагает defaults типа `<orig>-A` / `<orig>-B`, оператор переопределяет на semantic slug (`tickets-intake` / `tickets-execution`). Перерисовывает таблицу.
- **Слить с другой** — операнд: ≥2 фичи + new slug. Создаёт merged фичу, удаляет исходные.
- **Переименовать** — операнд: фича + new slug. Update без перераспределения функций.
- **Отменить разбиение** — отказ от flagged candidate, остаётся default boundary.
- **Подтвердить** — фича остаётся как есть (default).

**Batch operations** — оператор может сказать «разбить tickets как предложено и слить org с tasks» одной репликой → скилл применяет всё атомарно, потом одна обновлённая таблица с финальным состоянием.

**Sub-step 1.6 — Finalization.** Оператор: «подтверждаю» / «финально» / «всё ок» → feature map locked. Переход к Sub-phase 2.

**Принципы (P6 ARCHITECTURE applied):** feature boundary — продуктовое решение, не агентское. Silent autoclustering запрещён. Все split/merge candidates показываются явно с reasoning'ом.

### Sub-phase 2: Topological sort

**Sub-step 2.1 — Build feature-level dependency graph.** Алгоритм аналогичен module-level Mermaid (Operational Rule 8), но на feature-level:

1. Для каждой пары фич (A, B), A ≠ B: проверить, есть ли в спеке ≥1 функция фичи A с `Зависит от` ≥1 функции фичи B.
2. Если да → добавить edge `A → B` (один на пару, дедуплицируется до feature-level).

**Sub-step 2.2 — Kahn's algorithm:**

1. Start с фич без incoming edges (нет upstream).
2. При тае — alphabetical order (стабильно для diff).
3. Remove processed, repeat с newly-eligible.
4. Output: ordered list + per фича upstream/downstream/peer lists.

**Sub-step 2.3 — Tie-breaking visibility.** Если ≥2 фичи в одном «уровне» (нет относительной зависимости между ними) — отметка **«взаимозаменяемый порядок»** в roadmap.md (field `Взаимозаменяем с:` per фича). Появляется только при tie-break, иначе omit. Visibility принцип: оператор видит, что эти фичи можно реализовывать в любом порядке или параллельно.

**Sub-step 2.4 — Cycle detection.** Если после Kahn'а остались unprocessed фичи → inter-feature цикл → переход в Sub-phase 3 (cycle handling).

**Если цикла нет** → переход в Sub-phase 4 (bundle generation) с finalized topo order.

### Sub-phase 3: Inter-feature cycle handling

**Важное:** inter-feature цикл — genuine new finding (function-level Шаг 12.2 не ловит). Function-level deps могут быть ацикличными (например, `F-a-1 → F-b-2` и `F-b-3 → F-a-4` — разные функции в каждом направлении), но aggregation до feature-level даёт bidirectional edge `feature-a ↔ feature-b`.

**Detect dialog format** — единый для всех вариантов:

```
Найден цикл на уровне фич: feature-A ↔ feature-B.

Цепочка зависимостей-участников:
  F-a-1 (<Name>) → F-b-2 (<Name>)
  F-b-3 (<Name>) → F-a-4 (<Name>)

(на уровне функций граф ацикличен — разные функции в каждом направлении.
Цикл появился при aggregation до фич: A и B имеют bidirectional связи.)

Варианты резолва:
  1. Объединить фичи A и B в одну.
     Подходит если они — единая фича жизненного цикла, разводить нет смысла.
  2. Переместить функцию через границу. Кандидаты:
     - F-a-3 → feature-B (зависит от F-b-2, можно перенести)
     - F-b-1 → feature-A (зависит от F-a-4, можно перенести)
     Подходит если одна функция архитектурно ближе к соседней фиче.
  3. Изменить зависимость в спеке (focused recovery Шага 12.2).
     Подходит если связь по факту ошибочная.

Что выбираешь?
```

#### Variant 1 — Merge cycle features

A↔B → одна merged фича. Скилл предлагает new slug (или operator-suggested), оператор подтверждает. Feature boundary меняется только для участников цикла, остальные фичи не трогаются. Cycle устранён by definition.

**Подходит когда:** bidirectional deps указывают на tight coupling — A и B архитектурно одна фича реализации.

Не требует правки спеки. Возврат к Sub-phase 2 (topo sort повтор).

#### Variant 2 — Move function across feature boundary

Operator: «переместить F-a-3 из feature A в feature B». Скилл сначала предлагает **кандидатов перемещения** — функции, чьи deps участвуют в цикле (например, F-a-3 имеет F-b-2 как upstream — кандидат к перемещению в B). Operator выбирает.

После перемещения — re-build feature dep graph → re-run topo sort. Если цикл остался — repeat dialog.

Не требует правки спеки, только feature boundary. Возврат к Sub-step 2.1 (re-build graph).

#### Variant 3 — Fix function-level dep (focused mini-12.2)

Operator: «зависимость в спеке неправильная, надо править саму спеку».

Скилл запускает **focused Шаг 12.2 recovery loop**, ограниченный участниками цикла:

1. Показать full dep chain: `F-a-1 → F-b-2 → F-a-4 → F-b-3 → F-a-1`.
2. Operator выбирает резолв per existing 12.2 опции:
   - Убрать одну зависимость (указать какую — какая карточка, какой edge).
   - Объединить две функции в одну.
   - Extract общую часть в третью функцию-предка.
3. Скилл применяет правку — Edit tool на spec file (карточка функции в `## Фичи и функции`). Перед apply показывает «было → стало» по затронутой карточке. Operator confirm.
4. Скилл пере-прогоняет **полный** Шаг 12.2 на всём function-level графе:
   - Проверка-сирота
   - Разорванная замкнутость
   - Циклические зависимости
5. Если passing → re-build feature-level graph → re-run topo sort → если clean → continue в Sub-phase 4.
6. Если 12.2 failing после правки → стандартный 12.2 recovery loop (operator резолвит, может потребоваться несколько правок).

**Visibility constraints:**

- Спека остаётся `signed`, но содержимое изменилось. Скилл явно говорит: *«правка применена к карточке F-a-1, спека остаётся signed».*
- Это inline update mode под капотом, но не объявляется как полноценный update mode (не предлагает другие update операции типа split/merge функций — focused на резолв конкретного цикла).
- Edit `decomposition: skipped` marker не trigger'ит — мы внутри active decomposition phase.

**Relation к D14 / D18 invariants:** `signed` status allows content edits per D14 update mode mechanics. D18 frozen invariant применяется только к `shipped` спеком. Variant 3 — constrained slice of D14 mechanics, не violation D18.

### Sub-phase 4: Bundle generation

Для каждой фичи в топологическом порядке — генерация bundle файла с 8 элементами.

**Per-element derivation:**

| # | Элемент | Источник в спеке | Filter | Формат записи |
|---|---------|------------------|--------|---------------|
| 1 | Функции этой фичи | feature membership (Sub-phase 1) | — | F-id (Name) + § F-id ref + 1-line из Назначение |
| 2 | Upstream contracts | `Зависит от` функций фичи | функции **вне** фичи | F-id (Name) + § ref + 1-line summary + ссылка на upstream feature slug |
| 3 | Downstream consumers | reverse pass: кто зависит от наших | функции **вне** фичи | то же что upstream + ссылка на downstream feature slug |
| 4 | CC inheritance (relevant) | § Сквозные политики + `Применяется к` | CC где `Применяется к` includes ≥1 функция фичи ИЛИ «всем функциям» | **verbatim** full CC entry |
| 5 | Strategic forks resolved | § Стратегические развилки + `Связи:` | Sf где `Связи:` mentions ≥1 функция фичи (только RESOLVED) | **verbatim** full Sf entry |
| 6 | Backlog parked references | карточки фичи `Не делаем вообще` + backlog § Parked | semantic match parked entry → pointer; if no match — verbatim phrase без backlog ref | F-id (Name) + § backlog ref + 1-line reason + audit marker, ИЛИ verbatim text-only |
| 7 | User groups (relevant) | § Section 2 + `Доступно группам` функций фичи | groups mentioned by ≥1 функция фичи | **verbatim** group definition |
| 8 | Relevant scenarios | § Section 9 + scenario references | scenarios mentioning ≥1 функция фичи | **verbatim** scenario text |

**Принципы derivation:**

1. **Pointer для длинных, verbatim для коротких.** Function cards (длинные) — pointers. CC/Sf/groups/scenarios (короткие, 3-8 строк each) — verbatim. Дублирование function content в bundle создало бы SSoT violation (D17 spirit).
2. **Filter strictly by reference.** Bundle включает только то, что explicitly связано с функциями фичи через formal fields (`Зависит от`, `Применяется к`, `Связи:`, `Доступно группам`). Не «прогадать relevance» по семантике.
3. **Empty section visibility.** Если элемент пустой — секция всё равно присутствует в bundle с короткой placeholder фразой типа `(нет — функции этого блока не упоминаются в группах пользователей)`. Visibility принцип: оператор видит что скилл проверил, а не молча omit'нул.
4. **CC «всем функциям» — обязательно включается.** Если CC помечена `Применяется к: всем функциям` — включается в каждый bundle.
5. **Sf со status DEFERRED — не включается.** DEFERRED Sf'ы мигрированы в backlog § Deferred при Шаге 12.3. Bundle не reads backlog deferred section в v0.6.0.
6. **Parked references — edge case резолва.** Semantic LLM-suggestion match phrase из карточки фичи → backlog parked entry. Match success → pointer с audit marker `[matched semantically from carded phrase «<original text»]`. Match fail → verbatim phrase без backlog ref. Operator видит обе ситуации distinct, может проверить корректность match'а при review.
7. **Plain language pass.** После bundle сгенерён — гештальт-проверка по Operational Rule 11. Pointer'ы (`§ F-id-1`, `§ Section 4`) — стабильные идентификаторы, не переводятся. Free text inside bundle — на plain Russian.

**Размер per bundle:** типичный 30-60 строк. Pre-baked skeleton — `references/bundle-template.md`. Скилл заполняет skeleton по элементам.

### Sub-phase 5: Standing-policy candidates flag (pt 11 minimum)

После generation bundle файлов, **перед** написанием roadmap.md — single scan на кандидатов в standing policies.

**Алгоритм:**

1. Скан `§ Сквозные политики` спеки.
2. Per CC проверка:
   - Помечена ли как `Inherits: CC-X from <previous spec>` → уже standing inherited, пропускаем.
   - Если **нет** Inherits — LLM-suggestion на основе текста (Resolution + Pattern) про indicators cross-iteration характера: «применяется ко всем функциям системы», «обязательно для всех future features», «во всех итерациях продукта», «глобальное правило».
3. CCs без Inherits и с indicators → **кандидаты в standing**.

**Если кандидаты есть** → блок в конце roadmap.md:

```markdown
## Кандидаты в standing policies (опционально)

Эти сквозные политики потенциально применимы ко всем будущим итерациям продукта. Можно явно зафиксировать как standing в backlog § Standing cross-cutting policies — это сэкономит дублирование в спеках следующих итераций.

- **CC1.** Tenant data isolation.
- **CC3.** SLA timer.

Полный mechanism standing-policy auto-detection и migration — дизайн v0.7. Здесь — только flagging. Зафиксировать сейчас можно вручную в backlog.
```

**Если кандидатов нет** → секция полностью omitted из roadmap.md (не пишем «нет кандидатов» — это часто, нет нужды захламлять).

**v0.7 relation:** full standing-policy auto-detection + migration mechanism — в v0.7 design (не реализован, см. `docs/handoffs/2026-05-26-product-spec-v0.7-retrospective-improvements-design-handoff.md`).

### Sub-phase 6: Output artefacts (write files)

Атомарная запись всех файлов plan directory'я. Subdirectory создаётся per iteration.

**Directory structure:**

```
<project>/docs/specs/
  YYYY-MM-DD-<product-slug>-<iteration-slug>-product-spec.md
  <product-slug>-product-backlog.md
  <product-slug>-iteration-<N>-plan/                  ← создаётся в decomposition
    roadmap.md
    bundle-<feature-slug-1>.md
    bundle-<feature-slug-2>.md
    ...
```

**roadmap.md** — skeleton в `references/roadmap-template.md`. Содержит:
- Frontmatter (date, status, source spec, total features, total functions).
- Sequence section с per-feature entries (Назначение, Функции, Реализуем после, Дальше используется в, Взаимозаменяем с, Статус, ссылка на bundle).
- Optional standing-policy candidates section (если есть, из Sub-phase 5).

**Поле `Назначение` (feature-level) — derivation rule.** Скилл генерирует 1-2 строки plain-language summary на основе aggregation Назначений функций фичи + feature slug semantics. LLM-suggestion. **Operator review мандаторен** — после generation скилл показывает proposed feature Назначения в одной таблице, оператор подтверждает или переписывает per фичу. Продуктовое решение (как фича видится оператору-управленцу), не агентское autoselection.

**Поле `Статус`** — manual operator update в v0.6.0. Auto-callback от feature-spec → v0.6.3+. Initial value при generation — все «запланирован».

**bundle-`<feature-slug>`.md** × N — skeleton в `references/bundle-template.md`. Содержит per фича: header (iteration, source spec, feature functions list) + 8 секций per derivation table.

**Mermaid в roadmap.md — out of scope v0.6.0.** Ordered list + per-feature upstream/downstream fields достаточны для navigation. Optional Mermaid visualization (по аналогии с module-level Mermaid v0.5.0) — amendment v0.6.3+ при empirical signal что text-only sequence неудобен при ≥10 фичах.

**Frozen invariant plan directory'я (после write):**

- **Mutable в v0.6.0:** только поле `Статус:` в каждом feature entry roadmap'а (manual progress tracking).
- **Immutable в v0.6.0:** content roadmap'а (sequence, dependencies, descriptions, function lists). Все bundle файлы целиком. Названия feature slug'ов.

**Plan status:**
- `signed` — установлен при generation. Соответствует `signed` status спеки.
- `shipped` — flip синхронно со спекой (когда оператор говорит «iter X отгружена» — flip и спеки, и плана).

**Soft enforcement в v0.6.0.** Скилл не блокирует edits (нет файловой защиты — markdown файлы свободно edit'абельны). Frozen invariant фиксируется как guideline в этом разделе SKILL.md + в roadmap.md frontmatter как декларация status. Hard enforcement — v0.6.3+.

**Workaround для feature boundary change в v0.6.0** (re-decomposition mechanism не реализован):

1. Manual delete plan directory (`rm -rf <slug>-iteration-<N>-plan/`).
2. Re-trigger Entry B новой сессией → видит «спека signed + плана нет» → предлагает generate заново.
3. Decomposition phase runs from scratch на current spec state.

Explicit hack, не silent flow. Operator знает что делает. Re-decomposition mechanism как proper UX feature — v0.6.3+.

### После Sub-phase 6 (decomposition complete)

- Plan directory создан со всеми файлами.
- Оператору показано: *«План создан в `<path>`. Файлы: roadmap.md + N bundle файлов. Status: signed. Готов к feature-spec session — открывай bundle файл нужного блока и подавай как input в feature-spec.»*
- User Review Gate (отдельно от gate'а спеки): «Открой roadmap и bundle файлы. Если есть правки до закрытия — скажи». После confirm — session closes.

## Output Format

### Расположение файлов

- Spec: `<project>/docs/specs/YYYY-MM-DD-<product-slug>-<iteration-slug>-product-spec.md`
- Backlog: `<project>/docs/specs/<product-slug>-product-backlog.md`

(Конкретное расположение по умолчанию зависит от OQ15 file convention для семейства hi_flow — на момент imp v0.1.0 по умолчанию — плоский `docs/specs/`.)

### Skeleton

Полный skeleton спеки — `references/product-spec-template.md`. Полный skeleton бэклога — `references/product-backlog-template.md`. Worked example — `references/example-contact-tracker-mvp.md`. Скилл читает эти три файла на старте сессии для анкоринга формата output'а.

### Принцип PRD-as-standalone

Spec самодостаточна для контекста итерации. Бэклог читается отдельно для памяти между итерациями. ARCHITECTURE.md проекта читается отдельно для архитектурного контекста — не дублируется.

### Принцип plain language

Spec и бэклог адресованы продуктологу, не разработчику. По умолчанию — продуктовый русский. Англицизм допустим только если выполняется один из трёх критериев (см. memory `feedback_plain_language_conditional.md`):

1. Устоявшийся термин в индустрии (MVP, KPI, API, UX, ROI), у которого нет распространённого русского аналога.
2. Объективно компактнее или точнее русского при том же смысле, без потери нюанса.
3. Игра слов или специфический жаргон, теряющий смысл при переводе.

Чисто инженерный жаргон (extract-all, payload, throughput, idempotency, fallback) — переводится или раскрывается. Понятные продуктовые имена переменных (target_weight, current_weight) — нормально как структурные идентификаторы.

Контроль — гештальт-просмотр готового текста: читается ли это как русский с редкими оправданными вкраплениями, или как англо-русская мешанина? Опасная зона — кластеризация двух+ англицизмов в одном предложении при наличии русских эквивалентов.

### Адаптивная глубина

Шаги 1-3 (Описание → Группы → Задачи) адаптируют глубину по типу проекта из Шага 1.2:
- Личный продукт: 1 группа, тривиальный список задач.
- Стартап-MVP / open-source: 2-4 гипотетические персоны, гипотетические задачи.
- Внутренний инструмент / коммерческий: 3-7+ реальных ролей, матрица «группа × задачи», **buyer outcomes** в Шаге 1.4.

## Operational Rules — что скилл обеспечивает

1. **Только опрос сверху вниз.** Шаг 5 (вывод поддерживающих функций) идёт **из** пользовательских функций через продуктовый язык probe-таблицы D9, не через инженерный «нужен ли auth?». Скилл переводит инженерный вопрос в продуктовый эквивалент перед задаванием оператору.

2. **D2 трёхтестовый гейт для каждой функции-кандидата.** Independent shippability + atomic user job + negotiability. Провал любого теста → переразложить (split / merge / drop), не пускать в список функций.

3. **D5 boundary test для каждой развилки-кандидата.** Меняет состав функций или связи между ними? → стратегическая развилка (Section 5). Иначе — уровень фичи, не записывается (пойдёт в feature-spec).

4. **D17 асимметричная дисциплина указателей в бэклоге.** Committed-enabler 5-6 строк (Входит/Не входит), committed-domain 3 строки, parked полное содержимое (любой Type), deferred полное содержимое, rejected one-liner.

5. **D18 frozen invariant для shipped-спек.** Никаких правок. Переопределение через новую спеку, latest wins.

6. **Дисциплина эскалации к оператору (P6).** Не задавать оператору вопросы по детерминированным действиям (переходы статуса, проверки целостности, валидация формата, миграция в бэклог) — выполнять без вопроса. Не принимать решения за оператора в продуктовых вопросах — эскалировать.

7. **Закрытие требует резолва всех стратегических развилок.** OPEN-развилки недопустимы при closure — должны быть либо RESOLVED, либо DEFERRED (мигрируют в бэклог), либо OUT-OF-SCOPE.

8. **Mermaid в Section 4 — feature-level (historical naming: module-level), обязателен.** Ноды графа = фичи (Identity, Communications, Sales и т.п.), связи = обобщения inter-feature dependencies, выведенные из поля `Зависит от` в карточках функций. Function-level Mermaid out — даже при успешном рендере не работает как граф для product-spec (33+ нод теряют обзорность; feature-level — правильный уровень обзора для продуктового артефакта). Источник истины — Feature assignment (после Шага 4) + поле `Зависит от` в карточках. Регенерация LLM'ом при изменении фич или inter-feature deps. Детали (алгоритм mapping, расщепление категорий, syntax guarantees, splitting большого графа, second graph) — см. mini-section «Mermaid в Section 4 — конвенции» ниже.

9. **Living document scope.** Spec — living **внутри** одной итерации (промежуточный feedback от feature-spec → режим update). Бэклог — living **через** итерации.

10. **Правило эскалации.** При неоднозначности, неразрешимой из доступных данных — эскалировать оператору, не перезапускать upstream-скилл.

11. **Жаргон скилла остаётся внутри скилла.** Внутренние идентификаторы дизайна (D-N решений, CCP-N сквозных проверок, P-N принципов проекта, atomic / shippability / negotiability как термины, probe-таблица, P-NAME подполитики, Sf-N развилки) и кальки с английского (`enabler`, `domain feature`, `scaffolding`, `cardinality tag`, `premortem`) используются скиллом для внутренних рассуждений и cross-references в собственных артефактах (design spec, implementation reports, ARCHITECTURE.md). В **operator-facing** общении в ходе сессии — переводятся в продуктовый русский.

Silent mirror запрещён — нарушает D4 для будущих пользователей не-авторов.

   **Универсальное правило для терминов вне таблицы.** Если в operator-facing блоке всплывает внутренний термин, не покрытый таблицей — скилл переводит через короткое описание-«как для нового оператора», не оставляет как есть. Таблица — минимум, не исчерпывающий список.

## Mermaid в Section 4 — конвенции

Детали реализации Operational Rule 8. Pre-baked skeleton с placeholder'ами — в `references/product-spec-template.md` (агент наполняет, не изобретает структуру с нуля).

### Алгоритм mapping function-level deps → feature-level edges

(«module-level» в названии Mermaid секции preserved как technical descriptor of granularity — vs function-level — но nodes концептуально = фичи.)

1. Для каждой пары фич (A, B), где A ≠ B: проверить, существует ли в спеке ≥1 функция фичи A, у которой в поле `Зависит от` указана ≥1 функция фичи B.
2. Если да → добавить **один** edge `A → B` в граф (не один на каждую function-dep — дедуплицируется до feature-level).
3. **Тип edge** определяется характером зависимостей:
   - Если ≥1 функция-источник имеет active dependency (использует поведение / данные фичи B при работе) → solid arrow (`A --> B`).
   - Если **все** зависимости — passive policy application (фича B применяет сквозную политику CC к фиче A, оператор использования не активен) → dashed arrow (`A -.-> B`).
4. **Edge labels пусты по умолчанию.** Если оператор хочет семантику связи — выносится в текстовый paragraph под Mermaid'ом («Communications использует Identity для проверки прав на send»), не в edge label (см. syntax guarantees).

### Категории групп и расщепление

Три универсальные категории (см. эвристику классификации в разделе Feature assignment): **Infra / Passive / Domain**. Пустую группу можно опустить целиком — например, в личном продукте без сквозных политик нет Passive.

**Опциональное расщепление любой категории на под-группы (override per product).**

Любая из трёх категорий может быть расщеплена на под-группы при сработавшем продуктово-семантическом триггере. **Default — без расщепления** (одна группа на категорию).

Чаще всего расщепляется Domain:
- **Demo / sales-oriented продукт** — `Wow` (продающие фичи) / `Structure` (поддерживающие).
- **Платформенный продукт** — `Core` / `Add-ons`.
- **Two-sided marketplace** — `Supply-side` / `Demand-side`.

Реже — Infra или Passive:
- **Infra:** B2B-продукт с двумя качественно разными identity-системами (агенты vs клиенты) — `Identity (internal)` / `Identity (external)`. Federated SSO + local identity. Триггер — **качественное различие политик / контрактов**, не количество фич.
- **Passive:** Большой enterprise с разделением `Compliance / Audit` (GDPR retention, regulatory logging) vs `Operational monitoring` (notifications, alerts) — двумя разными областями ownership и политик.

**Где детектируется явное введение различения:**

1. **Operator dump** — самое раннее место. Оператор сам поднимает различение в свободной форме.
2. **Шаги 1-3** (описание продукта / группы пользователей / задачи) — типичный момент, когда продуктовая структура проясняется.
3. **Шаг 5 probes** — реалистичный поздний триггер. Probes могут выявить две auth-системы, что является сигналом к расщеплению Infra. Допустимо предложить расщепление **задним числом** после выявления в probes, до feature assignment'а финализации.

**Детекция — semantic LLM-суждение** на основе текста Operator dump'а и ответов в Шагах 1-3 / 5, не regex по ключевым словам и не количество фич. Признак различения — оператор называет в одной категории сущности с качественно разными политиками / контрактами / аудиториями.

**При обнаружении** скилл задаёт оператору прямой вопрос: «*Вижу различение [описание] в [категория]. Расщепить на под-группы [предлагаемые имена]?*». Оператор подтверждает / отказывается / переименовывает. Если уточнение пришло после feature assignment'а — скилл возвращается к нему для пересмотра групп без полного rerun Шагов 4-5.

**Одновременное расщепление двух категорий** допустимо. Если в продукте есть и две identity-системы (расщепление Infra), и compliance/operational split (расщепление Passive) — обе расщепляются. Скилл предлагает их параллельно (один общий блок вопросов), оператор подтверждает каждое решение отдельно.

**Skeleton при расщеплении** — см. фрагмент в `references/product-spec-template.md`. Расщеплённые под-группы — вложенные subgraph'ы внутри родительской категории либо две отдельные subgraph'ы без родительской обёртки (выбирает оператор). Производные classDef per под-группа — конкретные hex указаны в template (вариации синей для Infra-подгрупп, серой для Passive, зелёной для Domain).

### Syntax guarantees

Жёсткие требования к Mermaid output'у — без них рендеринг ломается:

- **Quoted labels.** Все node labels в двойных кавычках: `IM1["Module-A"]`. Без кавычек дефис ломает парсер (интерпретируется как минус-оператор).
- **ASCII edge labels или пустые.** Кириллица в edge labels не парсится в части viewer'ов: `IM1 --> DM1` (без label) или `IM1 -- "uses" --> DM1` (ASCII label). Если нужна семантика связи на русском — выносится в текстовый paragraph под Mermaid'ом, не в edge label.
- **Alphanumeric node IDs без дефисов / точек / пробелов.** ID ноды — `IM1`, не `F-id-1` (дефис) и не `F.id.1` (точка). Реальные F-ID функций — внутри label если нужны, но в IDs нод они не идут.
- **≤20-25 нод в одном графе.** Feature-level (т.н. «module-level» в historical naming) обычно даёт 8-15 нод (комфортно). Если больше — см. splitting ниже.

### linkStyle fragility — honest constraint

`linkStyle` индексируется порядком объявления edges. При вставке нового edge в середину — все последующие `linkStyle` съезжают на 1. При ручной правке Mermaid в спеке оператор должен либо добавлять edge в конец списка (тогда добавить новый linkStyle тоже в конец), либо просить скилл регенерировать весь linkStyle блок целиком. Скилл при регенерации перестраивает блок целиком, не пытается point-update.

### Splitting при >25 фичах

1. Использовать категориальную группировку (Infra / Passive / Domain) в subgraph wrappers, оставляя ноды на feature-level — обычно достаточно, потому что внутри subgraph фич не больше 6-8.
2. Если внутри одной категории ≥4 фич с тонкими взаимосвязями (вторичные связи не критичны для product-level overview) — объединить в **superфичу** (одна нода с агрегированным именем «Sales superфича: Pipeline + Quotes + Contracts + Invoices»). Связи к superфиче аналогично — один edge на категорию.
3. Если даже категориальная группировка + superфича не вмещается в 25 нод — выделить второй feature-level граф для отдельного flow (см. second graph ниже).
4. Traceability при объединении в superфичу — комментарий под графом: «Sales superфича = F-sales-*, F-quote-*, F-contract-*, F-invoice-* (см. карточки в Section 4)». Поле `Зависит от` в карточках остаётся на feature-level, не агрегируется.

**Порядок при коллизии splitting × расщепление категории.** Если одновременно срабатывают триггеры расщепления категории (продуктовая семантика) и superфича-компрессии (≥4 фич с тонкими взаимосвязями) — **расщепление применяется первым** (отражает семантику), superфича — внутри получившейся под-группы (механическая компрессия). Комбинация допустима; внутри расщеплённой под-группы superфича выглядит как одна агрегированная нода с тем же стилем, что и под-группа.

### Опциональный second graph

Если в спеке есть критичная cross-functional цепочка (lead → contract → soft approval → contract active в CRM) — под неё может быть дополнительный mini-граф под основным feature-level. Обозначается explicit заголовком: `### Mermaid: Cross-functional flow — <название цепочки>`.

**Триггер для проактивного предложения** скиллом:
- Цепочка ≥4 функций
- Проходит через ≥3 разных фичи
- Имеет ≥1 soft transition (требует подтверждения оператора между шагами; не автоматический pipeline)

Если все три условия выполнены — скилл проактивно предлагает: «*Вижу в спеке критичную cross-functional цепочку <название>. Добавить mini-graph под основным feature-level?*». Оператор подтверждает или отказывается. Если оператор сам просит second graph для какой-то цепочки — добавляется независимо от триггеров (override).

Эта возможность — расширение, не default; включается только при сработавшем триггере или явной просьбе.

## Format Rules

1. **Иерархические ID (Cockburn-style).** Функции: `F-<feature-slug>-N` (например, F-id-1, F-comm-1, F-sales-2). Стратегические развилки: `Sf1, Sf2 ...`. Сквозные политики: `CC1, CC2 ...`. Переиспользуемые подполитики: `P-<UPPER-CASE-WITH-DASHES>`.

2. **ID назначен ⇔ карточка существует.** Нет висящих ID без карточек. Нет карточек без ID.

3. **Поля `Тип` и `Статус` обязательны на каждой карточке функции.** Тип ∈ {пользовательская, поддерживающая}. Статус ∈ {in-scope, parked, out-of-scope}.

4. **Тег кардинальности (XOR / OR / OPT)** обязателен на стратегических развилках с ветвями.

5. **`Зависит от`** — обязательное поле для **всех** функций (для domain-функций часто пусто или ссылки на enabler'ы; для enabler'ов часто пусто или ссылки на другие enabler'ы). Mermaid строится по нему.

6. **`Inherits`** — опциональное поле в карточке для standing-политик из предыдущих итераций.

7. **`Доступно группам`** — обязательное если Шаг 9 сработал и группы различаются. Иначе опускается.

8. **Все стратегические развилки при closure** — RESOLVED / DEFERRED / OUT-OF-SCOPE. OPEN-развилки при closure — блокирующая ошибка.

9. **Нет резюме в конце спеки** — оператор читает в полном объёме по принципу PRD-as-standalone.

10. **Имя файла бэклога** — `<product-slug>-product-backlog.md` (без даты, один на продукт).

11. **Имя файла спеки** — `YYYY-MM-DD-<product-slug>-<iteration-slug>-product-spec.md`.

12. **Имя plan-директории (v0.6.0)** — `<product-slug>-iteration-<N>-plan/` рядом со спекой. Внутри: `roadmap.md` + `bundle-<feature-slug>.md` × N. Feature slug'и — стабильные kebab-case identifiers (по умолчанию = feature-slug из Feature assignment, при splits — operator-defined типа `tickets-intake` / `tickets-execution`).

13. **Frozen invariant для plan directory (v0.6.0)** — после generation plan'а вся content immutable, кроме поля `Статус:` в каждом feature entry roadmap'а (manual progress tracking). Bundle файлы целиком immutable. Изменение feature boundary после sign-off — explicit workaround: manually delete plan directory + re-trigger Entry B (re-generation на current spec state). Re-decomposition mechanism как proper UX — v0.6.3+. Soft enforcement в v0.6.0 (файловой защиты нет, фиксируется как guideline в SKILL.md + roadmap frontmatter).

15. **F-ID convention (v0.6.1).** `F-<feature-slug>-<N>` parses естественно как «функция N внутри фичи `<feature-slug>`». F-* префикс preserved для backward compatibility с pre-v0.6.1 artefacts. Generic «functional element» framing — `F-` covers как individual function reference, так и подразумеваемую parent фичу через slug.

14. **Метаданные spec (v0.6.0)** — frontmatter может содержать optional поле `decomposition: skipped` (для spec'ов, по которым оператор явно отказался от decomposition phase per Entry A option 3). Маркер ставится только на `signed` спеках (на `shipped` запрещено per D18 frozen invariant). Existence маркера suppresses Entry B auto-prompt в session setup.
