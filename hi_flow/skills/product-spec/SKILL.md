# `hi_flow:product-spec` — Product-Level Decomposition Skill

Помоги оператору превратить продуктовую идею (или новую итерацию существующего продукта) в замороженный `product-spec.md` для одной отгружаемой итерации плюс живой `product-backlog.md`, который накапливает реестр функций между итерациями.

Скилл системно вытаскивает пользовательские (domain) функции и производные поддерживающие (enabler) функции через структурированный опрос на продуктовом языке (сверху вниз per D9), классифицирует стратегические развилки по тесту границ D5, управляет памятью между итерациями через бэклог с асимметричной дисциплиной размера записей.

## Out of scope

- Углубление в одну фичу — отдельный скилл `hi_flow:feature-spec`.
- Архитектурные решения — `hi_flow:arch-spec` (выводит архитектуру из готовых product-spec + feature-spec).
- Implementation planning — `hi_flow:impl-plan` припаркован; покрывается Superpowers TDD methodology.
- Auto-invoking следующей фазы — оператор инициирует переход вручную.

## Concepts

Базовые понятия, на которых стоит вся таксономия скилла. Если оператор работает с product-spec впервые — это словарь.

- **Trinity функций (D1).** Любая функция в продукте — одна из трёх категорий: **пользовательская (domain feature)** даёт ценность юзеру напрямую (контакты, заявки, отчёты); **поддерживающая (enabler feature)** обслуживает пользовательские, но содержит продуктовые развилки (auth, права, нотификации); **scaffolding** — чисто инженерные решения без продуктовых развилок (схема БД, deployment, CI/CD, бэкапы). Product-spec фиксирует только domain + enabler. Scaffolding — материал arch-spec, в product-spec не записывается.
- **Модуль = кластер, не уровень иерархии (D3).** Заголовок «### Модуль: <Name>» в Section 4 — организационная метка для группировки родственных функций (Identity, Contacts, Leads), а не уровень декомпозиции. Функции внутри модуля равноправны и связаны через DAG, не через вложенность. Модуль не имеет собственной карточки и собственных полей.
- **DAG зависимостей (D4).** Связи `Зависит от` между функциями образуют направленный ациклический граф. Циклы — блокирующая ошибка целостности (см. Шаг 12.2). Порядок функций в backlog § Shipped и при closure определяется топологическим сортом DAG'а: зависимости впереди зависимых.
- **Одна спека = одна итерация (D16).** Product-spec фиксирует ровно одну отгружаемую итерацию. Никаких «v2 parked в этой же спеке» или «v3 roadmap внизу». Будущие итерации живут в backlog'е (parked / deferred), а сама v2 — это отдельная новая product-spec через несколько месяцев. Latest wins (D18).

## Output

**Два артефакта:**

1. **product-spec.md** — замороженная запись процесса дизайна одной итерации. Default location: `<project>/docs/specs/YYYY-MM-DD-<product-slug>-<iteration-slug>-product-spec.md` (configurable per OQ15).

2. **product-backlog.md** — реестр функций, живущий через все итерации одного продукта. Default location: `<project>/docs/specs/<product-slug>-product-backlog.md` (configurable). Создаётся при первой итерации, обновляется в каждой последующей.

## Anti-triggers (do NOT auto-activate)

- «спека на фичу X», «продуктовый дизайн фичи X» — это уровень feature-spec, не product-spec. Предложить правильный скилл, не запускать product-spec.
- «давай добавим функцию Y в продукт» — может быть либо feature-spec (если продукт уже существует), либо update mode product-spec (если меняем scope текущей итерации). Уточнить у оператора, не запускать автоматически.
- «спроектируй архитектуру для X» — территория arch-spec.
- «реализуй X», «напиши код» — фаза имплементации.
- «расскажи про методологию» — research / explanation, не запуск скилла.

## Mode

Скилл работает **только через brainstorm-путь**. Direct-путь не применяется на продуктовом уровне — продукт по природе многофункциональный, многосегментный, и вывод поддерживающих функций сверху вниз (D9) теряется при пропуске опроса.

**Никаких самооценочных предложений оператору на старте сессии.** Скилл сразу переходит к Process flow.

Если оператор явно скажет «direct» / «без проб» — отказать со ссылкой на D13 и предложить короткий brainstorm: тривиальные шаги в простых продуктах пробегаются быстро, но проходятся все.

## Process flow

Сессия проходит через четыре фазы:

1. **Session setup** — определение активной спеки и загрузка бэклога.
2. **Operator dump** — открытое описание продукта оператором без структурирования.
3. **Agent probing** — двенадцатишаговая таксономия (см. ниже), последовательно. Сквозные проверки активны параллельно.
4. **Closure** — Шаг 12 (подтверждение scope + проверка целостности + миграция в бэклог при закрытии) + завершающий премортем + таблица открытых вопросов.

После closure — User Review Gate, потом передача в feature-spec (если оператор инициирует) или закрытие сессии.

### Session setup

При активации скилла:

1. **Glob-проверка** в настроенном расположении (по умолчанию `<project>/docs/specs/`):
   - `*product-spec.md` — существующие файлы product-spec.
   - `<product-slug>-product-backlog.md` — бэклог текущего продукта.

2. **Определить активную спеку** среди найденных. Активная = поле `status` в метаданных ∈ {draft, signed}. Замороженная = `status: shipped`.

3. **Ветвление по результату:**
   - **Активная спека найдена** → предложить режим update: «Найдена активная спека [path], status [status]. Продолжаем работу над ней?». По умолчанию — да; оператор может переопределить («закрой active, начнём новую»).
   - **Спек нет** → первая итерация продукта, с нуля без переопределений.
   - **Только замороженные спеки** → предложить новую спеку для следующей итерации: «Найдены замороженные спеки [list]. Стартуем новую итерацию?». По умолчанию — да.
   - **Запрос на переоткрытие замороженной спеки** («открой повторно X для дополнения») → отказать: «Замороженные спеки не редактируются (D18 frozen invariant). Если решение из iter1 нужно пересмотреть — это новое решение в новой спеке, latest wins».

4. **Бэклог загружается в контекст** для переиспользования поддерживающих функций между итерациями и автоподсказок из parked.

5. **Подтверждение оператора** (явное или через продолжение работы) → переход в фазу Operator dump.

### Operator dump

Открыть единым широким вопросом:

> *«Расскажи про продукт. Что это, для кого, какие задачи решает. Дамп всё, что в голове.»*

Если активная спека / режим update — формулировка адаптируется:

> *«Что меняется в этой итерации продукта по сравнению с уже зафиксированным? Что нужно добавить, что убрать, что переосмыслить?»*

Скилл слушает, не задаёт уточняющих вопросов. Оператор может быть кратким или подробным — обе ситуации валидны.

Дамп может быть пустым («хочу X, ничего пока не думал») — переходим к опросу без задержки.

## Probing Taxonomy

Floor checklist для brainstorm path: **6 базовых обязательных (Шаги 1-6) + финальный обязательный (Шаг 12) = 7 обязательных всего + 3 условных + 2 опциональных + 3 сквозных + 1 завершающая**.

### Universal rule

Пройти все категории, но **не обязательно** генерировать output в каждой условной/опциональной — явный N/A допустим для тех, у которых триггер не сработал. Обязательные шаги дают output всегда.

**Адаптивная глубина** — глубина шагов 1-3 варьируется по типу проекта (личный / стартап-MVP / open-source / внутренний инструмент / коммерческий). Шаги 4-12 одинаковы независимо от типа.

Каждый шаг имеет процедуру (вход → алгоритм → выход). Адаптивные уточняющие вопросы допускаются как **дополнение** к процедуре, не замена.

### Обязательные шаги

#### Шаг 1. Описание продукта

**Процедура:**
- 1.1. Спросить: «Что это за продукт — одной фразой?»
- 1.2. Уточнить тип проекта: личный / open-source / стартап-MVP / внутренний инструмент / коммерческая разработка под заказчика.
- 1.3. *Если проект не личный:* «Кто заказчик (платит) и кто пользователь (использует)? Это разные роли?»
- 1.4. *Если коммерческий или внутренний:* «Какие бизнес-цели у заказчика?» — это драйверы приоритизации, не сами функции.
- 1.5. *Если стартап-MVP или open-source:* «Какие гипотетические типы пользователей? Обоснуй.»

**Output anchor:** Section 1 (Описание продукта) спеки.

#### Шаг 2. Группы пользователей

**Процедура:**
- 2.1. Перечислить группы или роли пользователей.
  - Личный продукт: 1 группа (фиксируем явно, даже если очевидно).
  - Стартап-MVP / open-source: 2-4 гипотетические персоны.
  - Коммерческий / внутренний: 3-7+ реальных ролей внутри организации заказчика.
- 2.2. Для каждой группы — описание роли и основная цель использования продукта.

**Output anchor:** Section 2 (Группы пользователей).

#### Шаг 3. Задачи каждой группы

**Процедура:**
- 3.1. Для каждой группы — список задач, которые её пользователи хотят решать через продукт.
- 3.2. *Если групп больше одной:* пересечения задач между группами (где одна группа создаёт что-то, другая потребляет).
- 3.3. *Если коммерческая разработка:* связь задач пользователей с бизнес-целями заказчика — какие задачи продвигают какие бизнес-цели (для приоритизации).

**Output anchor:** Section 3 (Задачи пользователей).

#### Шаг 4. Перечисление пользовательских функций

Пользовательская функция = то, что юзер видит и использует, что само по себе даёт ему ценность.

**Процедура:**
1. Каждая задача из Шага 3 → кандидат в пользовательскую функцию.
2. Применить три теста D2:
   - **Independent shippability** — отгрузить только это, будет ли value? Если ответ «нет» — не фича, переразложить.
   - **Atomic user job** — один user job-to-be-done на функцию? Если внутри три разных job — псевдо-модуль, разбить.
   - **Negotiability of inclusion** — «делаем ли X?» осмысленный продуктовый вопрос?
3. Кандидат, не прошедший хотя бы один тест → переразложить (раздробить, объединить, отбросить).
4. **Автоподсказка из parked-бэклога.** После описания оператором каждого кандидата — скилл сканирует parked-бэклог на семантическое пересечение. Совпадение → подсветить оператору: «уже было запарковано похожее F-X из iterN, level [level]. Поднять / слить / держать отдельно?». Эвристика совпадения — семантическая (LLM-суждение), не regex. Признаки совпадения: близкие названия, перекрывающиеся границы (Назначение / Входит), одинаковые upstream-зависимости.
5. Результат: список пользовательских функций с одной строкой описания на каждую.

**Output anchor:** Section 4, карточки с `Тип: пользовательская`.

#### Шаг 5. Вывод поддерживающих функций (сверху вниз)

Поддерживающая функция = то, что нужно, чтобы пользовательские функции работали (учётная запись, права, восстановление пароля, нотификации, поиск, журнал действий, экспорт). У них тоже есть продуктовые решения, поэтому они в продуктовой спеке.

**Процедура:**
1. Для каждой пользовательской функции — пройти список вопросов в продуктовом языке (probe table per D9):

| Инженерный вопрос (НЕ задавать) | Продуктовый эквивалент (задавать оператору) | Поддерживающая функция, которая выводится |
|---|---|---|
| Нужен ли auth? | Один тип юзера или разные роли? | User CRUD |
| Какой auth method? | Самостоятельная регистрация или invite-only? | Registration flow |
| RBAC или ABAC? | Все админы равны или есть градации прав? | Permissions model |
| Multi-tenancy? | Несколько изолированных компаний делят инстанс? | Tenancy model |
| Какой password reset? | Юзер может забыть пароль и восстановить сам? | Password reset flow |
| Нотификации? | Юзер хочет узнавать о событиях, когда он не в системе? | Notifications channel(s) |
| Search infra? | Юзеру нужно найти запись среди многих? | Search enabler |
| Audit log? | Нужно знать, кто что когда сделал? | Audit log enabler |

2. Каждое продуктовое «да» порождает кандидата на поддерживающую функцию.

3. **Проверка shipped-бэклога (переиспользование поддерживающих функций между итерациями).** Перед созданием новой enabler-карточки скилл проверяет shipped-бэклог на совпадение по указателю (Входит / Не входит). Дерево решений:
   - **Совпадение (Входит покрывает нужное)** → предложить **переиспользование**: «У тебя F-id-2 (Permissions) shipped. Покрывает плоские роли admin/user, что нужно. Используем без новой карточки?». Оператор подтверждает → новая domain-карточка получает `Зависит от: F-id-2`, новый enabler не создаётся.
   - **Частичное совпадение (Входит покрывает основное, но нужно добавить)** → предложить **расширение**: «F-id-2 покрывает плоские роли, но не иерархические. Создаём F-id-2-ext в этой итерации?». Оператор подтверждает → новая карточка F-id-2-ext, зависит от F-id-2.
   - **Несовпадение (Не входит явно содержит нужное)** → предложить **новую карточку**: «F-id-2 не покрывает ABAC. Создаём F-id-3 (ABAC permissions)?».
   - **Неоднозначно (указателя недостаточно)** → скилл **читает исходную спеку по требованию** для полной карточки enabler'а, делает финальное решение.

4. **Автоподсказка из parked-бэклога** (если shipped-проверка не дала совпадения). Скилл сканирует parked-бэклог на семантическое совпадение — совпадение → подсветить оператору, как в Шаге 4.

5. Результат: список поддерживающих функций с зависимостями обратно к пользовательским, которым они служат.

**Output anchor:** Section 4, карточки с `Тип: поддерживающая`.

#### Шаг 6. Границы каждой функции

Для каждой функции (пользовательской и поддерживающей) заполнить три поля:

- **Входит:** что включаем в эту функцию в этой версии.
- **Не входит (но это отдельная функция):** что вынесено в другую функцию роадмапа.
- **Не делаем вообще:** что отрезано на уровне продукта.

**Процедура:**
1. Уточняющие вопросы оператору: «Какой минимально полезный кусок этой функции? Где её разумная граница? Что точно не делаем в этом продукте?»
2. **Отметка о совпадении при парковке.** При обсуждении «выносим в parking» — скилл сканирует parked-бэклог на семантическое пересечение. Совпадение → подсветить: «уже было запарковано похожее F-X из iterN, слить или держать отдельно?».
3. Эвристика для оператора: если в product-spec нужно расписать функцию больше 5-8 строк, чтобы решить «включаем в MVP» — детализация ушла глубоко. Подняться обратно до scope cut. Дальнейшее — feature-spec.

**Output anchor:** Section 4 карточки, поля Входит / Не входит / Не делаем вообще.

### Условные шаги (запускаются при срабатывании триггера)

#### Шаг 7. Сквозные политики — inheritance-aware

**Триггер:** ≥2 in-scope функций в этой итерации **ИЛИ** новая функция вводит сквозную политику (audit log, нотификации, поиск, экспорт, файловое хранилище и т.п.), которой не было в предыдущих итерациях.

**Процедура:**
- 7.1. Скилл читает бэклог § Standing policies + спеки предыдущих итераций — список применимых уже-известных политик.
- 7.2. Для каждой in-scope функции этой итерации скилл проверяет: попадает ли под существующие standing-политики? Если да — функция **наследует политику** (в карточке добавляется поле `Inherits: CC-X from <previous spec>`).
- 7.3. Если в текущей итерации появляются новые сквозные политики, **не покрытые** существующими — задать вопрос оператору («Единый audit log поверх всех функций или у каждой свой?», «Общая система нотификаций или каждая функция шлёт свои?»). Output: новая CC-политика в spec § Сквозные политики, потенциально новая поддерживающая функция (общий сервис журнала / общий узел нотификаций).

**Если все сквозные политики этой итерации покрыты существующими standing-политиками — Шаг даёт результат «только унаследовано, новых политик в спеке нет».** Это валидное завершение, не пропуск.

**Output anchor:** Section 6 (Сквозные политики).

#### Шаг 8. Жизненный цикл данных — inheritance-aware

**Триггер:** in-scope функция этой итерации хранит постоянные данные **И** либо нет существующих политик жизненного цикла (первая такая функция в продукте), либо новая функция вводит аспект жизненного цикла, не покрытый существующими.

**Процедура:**
- 8.1. Скилл читает существующие политики жизненного цикла (если есть в бэклоге или предыдущих спеках).
- 8.2. Для каждой in-scope функции, которая хранит постоянные данные — проверка: существующие политики хранения / GDPR / миграции покрывают?
- 8.3. Если новая функция вводит новый аспект жизненного цикла данных — задать вопрос («Что происходит с данными юзера, когда он уходит?», «Как долго хранить логи?», «Нужны ли право на забвение и экспорт?», «Soft vs hard delete?»). Output: новая политика жизненного цикла в спеке этой итерации, возможно новая поддерживающая функция (сервис retention, экспорт GDPR), стратегические развилки.

**Если все аспекты жизненного цикла покрыты существующими политиками — Шаг даёт результат «только унаследовано».** Аналогично Шагу 7.

**Output anchor:** Section 6 (Сквозные политики), часть про lifecycle.

#### Шаг 9. Взаимодействия между группами пользователей

**Триггер:** групп больше одной (Шаг 2 даёт ≥2).

**Процедура:** задать оператору: «Какие сценарии требуют участия двух и более групп? Кто видит данные кого? Какие функции доступны только определённым группам?»

**Output anchor:** Поле `Доступно группам` в карточках Section 4; развилки в межгрупповых сценариях — Section 5.

### Опциональные шаги (только при явном основании)

#### Шаг 10. Соответствие нормативным требованиям

**Триггер:** домен подлежит регулированию (медицина, финансы, образование, обработка персональных данных GDPR, доступность WCAG).

**Процедура:** задать оператору: «Какие требования законов или стандартов применимы? Кого они затрагивают? Какие функции продукта обязаны им соответствовать?»

**Не выдумывать требования для нейтральных доменов.** Если триггер не сработал — явный N/A.

**Output anchor:** Жёсткие политики в Section 6.

#### Шаг 11. Базовые требования к нагрузке и доступности

**Триггер:** оператор поднимает явно либо домен высокоставочный.

**Процедура:** задать оператору: «Порядок числа пользователей? Ожидания по скорости отклика? Гарантии доступности? Объём данных? Работа без интернета или только онлайн?»

В продуктовую спеку попадает только то, что заказчик / оператор осознанно выбирает как продуктовое решение (например: «обязательно работает офлайн»). Конкретные числа нагрузки — материал арх-спеки.

**Output anchor:** Section 8 (Базовые требования к нагрузке и доступности).

### Финальный обязательный шаг

#### Шаг 12. Подтверждение scope + проверка целостности + миграция в бэклог

Запускается после Шагов 4-11 (всё содержание итерации зафиксировано).

**Процедура:**

- **12.1. Подтверждение scope.** Скилл показывает оператору финальный сводный список: in-scope функции (с карточками), parked-в-этой-итерации (с level), deferred-развилки, rejected. Оператор подтверждает или правит.

- **12.2. Проверка целостности** (скилл выполняет автоматически):
  - **Проверка-сирота** — enabler в in-scope, ни одна domain-функция от него не зависит → подсветить: «F-X в scope, никто не использует — точно нужен?».
  - **Разорванная замкнутость** — domain-функция в in-scope, её enabler в parked / out-of-scope → подсветить: «F-Y зависит от F-Z, который parked — резолви».
  - **Циклические зависимости** — цикл в графе зависимостей → блокирующая ошибка с требованием резолва.

- **12.3. Миграция в бэклог при закрытии:**
  - in-scope → бэклог § Shipped (асимметричные указатели по типу — см. Backlog mechanics ниже).
  - parked-в-этой-итерации → бэклог § Parked (полное содержимое + поле level).
  - deferred-развилки → бэклог § Deferred (полное содержимое).
  - rejected → бэклог § Out-of-scope (one-liner).
  - resolved-развилки остаются в spec § Стратегические развилки.
  - Iteration index в бэклоге обновляется новой записью.

- **12.4. Переход статуса.** При подтверждении оператора — статус спеки автоматически переходит `draft` → `signed` (готов к feature-spec). Переход `signed → shipped` — позже, по явной команде оператора («iter X отгружена / iter X в production»). Скилл применяет переходы детерминированно, без эскалации (см. Operational Rules).

**Output:** spec готова к User Review Gate, бэклог обновлён, frozen invariant сохраняется.

### Сквозные проверки (постоянно активны во время всей сессии)

#### CCP1. Выявление неявных критериев

Если в ответе оператора всплывает расплывчатое слово (быстро, часто, много, опасный, похоже, естественно) — задать уточняющий вопрос: переводим в число, в правило-чеклист, или принимаем как принципиально нечёткое и закрепляем якорным примером.

#### CCP2. Проверка на противоречия

После каждого нового ответа сверять с уже записанной структурой. Если новый ответ противоречит чему-то — поднять конфликт оператору, не записывать молча:

> *«В Шаге 4 ты сказал X, в текущем ответе предполагается Y. Конфликт. Какое решение верно / нужно ли пересмотр?»*

#### CCP3. Классификация стратегических развилок

Когда в ответах всплывает развилка-кандидат — применять **тест D5**:

1. Меняет ли feature list? Ветки развилки приводят к разным спискам функций в роадмапе?
2. Меняет ли cross-feature deps? Ветки реорганизуют граф зависимостей (новые связи, новые корни)?

Любое «да» → стратегическая развилка → секция 5 спеки. Если оба «нет» — feature-уровень, не записывается в продуктовую спеку, пойдёт в feature-spec.

В серых зонах оператор решает явно («это будет product-spec fork» / «это уйдёт в feature-spec»). Скилл подсвечивает серую зону, не выбирает за оператора.

**Автопроверка по deferred-развилкам.** Также проверка пересечения кандидата-развилки с deferred-развилками из бэклога. Совпадение → подсветить оператору: «связано с deferred Sf-Y из iterN, резолвить сейчас или продолжать как отдельную развилку?».

Перед завершением сессии — отдельный финальный проход по всем накопленным развилкам для подтверждения корректной классификации.

### Завершающая проверка — премортем на 6 месяцев

В конце сессии задать вопрос:

> «Представь — продукт запущен, прошло 6 месяцев. Что пошло не так? Сценарии: пользователи не пришли / пришли но ушли / конкурент съел рынок / технический долг сломал скорость разработки / сам оператор отказался поддерживать продукт.»

Каждое представимое осложнение → потенциально новая функция, стратегическая развилка или граница, которую упустили.

**Output anchor:** Section 11 (Что может пойти не так).

### Условие завершения сессии

Перейти к закрытию можно только когда:

- Все обязательные шаги (1-6, 12) дали результат.
- Все условные шаги (7-9) с сработавшими триггерами дали результат либо явно помечены «не применимо».
- CCP3 прошла финальный проход.
- CCP2 — нет открытых конфликтов.
- Премортем проведён.

## Backlog mechanics

Бэклог хранит реестр функций, живущий через все итерации продукта, с **асимметричной дисциплиной размера записей** (защита от неограниченного роста по уроку из архитектурного документа проекта).

### Содержание бэклога по типам записи

**Iteration index** — таблица с одной строкой на итерацию:

| Дата | Slug | Status | Spec |
|------|------|--------|------|
| 2026-05-10 | erp-mvp | shipped (2026-07-15) | docs/specs/2026-05-10-erp-mvp-product-spec.md |

**Указатель на shipped-enabler** (5-6 строк):

```markdown
### F-id-2. Permissions [shipped iter1: erp-mvp]
**Module:** Identity
**Type:** enabler
**Назначение:** базовая модель прав доступа на уровне ресурсов
**Входит:** flat-роли (admin/user/viewer); per-resource permissions; assign/revoke/list
**Не входит:** иерархические роли с inheritance; ABAC; audit log
**Spec:** docs/specs/2026-05-10-erp-mvp-product-spec.md § F-id-2
```

**Указатель на shipped-domain** (3 строки):

```markdown
### F-comm-1. Email-conv [shipped iter1: erp-mvp]
**Module:** Communication
**Type:** domain
**Назначение:** двусторонние email-разговоры оператора с клиентами с threading
**Spec:** docs/specs/2026-05-10-erp-mvp-product-spec.md § F-comm-1
```

**Parked-функция** (полное содержимое + поле level):

```markdown
### F-comm-X. Auto-classification (level: detailed)
**Status:** parked
**Originating analysis:** docs/specs/2026-08-15-erp-multichannel-product-spec.md (Секция 4)
**Reason for parking:** ML-стоимость не вписалась в бюджет iter2
**Carry-over candidate for:** iter3+
[full card content]
```

Уровни parked: `detailed / partial / note / fragment`. Оператор выбирает level при парковке (скилл спросит: «насколько детально хочешь сохранить?»).

**Deferred-развилка** — полная карточка как было в спеке + причина откладывания + ссылка на исходную спеку.

**Out-of-scope (rejected)** — one-liner: `F-Z. Mobile app native — отвергнуто iter1, причина: scope MVP не выдерживает; альтернатива — responsive web.`

**Standing-политики** — указатель (1 строка + ссылка на исходную спеку).

### Асимметричная дисциплина указателей — обоснование

| Тип записи | Формат | Размер |
|---|---|---|
| Указатель на shipped-enabler | ID + Name + Module + Type + Назначение + Входит + Не входит + spec ref | 5-6 строк |
| Указатель на shipped-domain | ID + Name + Module + Type + Назначение + spec ref | 3 строки |
| Parked-функция (любой Type) | Полный контент + level + причина | varies |
| Deferred-развилка | Полная карточка + причина откладывания | varies |
| Out-of-scope rejected | One-liner + причина | 1 строка |
| Запись в iteration index | Дата + slug + статус + spec ref | 1 строка |
| Указатель на standing-политику | ID + Name + spec ref | 1 строка |

**Почему асимметрия.** Shipped растёт линейно во времени без верхней границы (каждая итерация добавляет, ничего не убывает) → должны быть указатели. Parked / deferred — ограничены по смыслу (поглощаются в новых итерациях по мере резолва) → могут быть полным содержимым. Domain shipped vs enabler shipped: переиспользование enabler'ов требует совпадения scope (нужны Входит / Не входит), переиспользование domain'ов чаще решается на уровне названия → разные форматы указателей.

### Поток закрытия (operational)

При Шаге 12.3:

1. Если бэклог не существует — создать с шапкой по skeleton'у `references/product-backlog-template.md`:

```markdown
# <Product> — Backlog

[Метаданные: дата создания, дата последнего обновления, версия структуры]

## Iteration index
(пусто на старте)

## Shipped features
(пусто на старте)

## Parked features
(пусто на старте)

## Deferred strategic forks
(пусто на старте)

## Out-of-scope (rejected)
(пусто на старте)

## Standing cross-cutting policies
(пусто на старте)
```

2. Миграция элементов в соответствующие секции (по § Миграция в бэклог в Шаге 12.3).
3. Iteration index обновляется новой записью в табличной форме.
4. Standing-политики — только для впервые введённых. Скилл подсвечивает «новая CC, добавить в standing?» при закрытии, если в спеке есть CC, ещё не помеченная как унаследованная. Триггер автоматического перевода в standing: в спеке iter2 вводится та же политика (по смыслу), что была в iter1 — скилл предложит «эта политика переезжает в standing, дублировать в каждой спеке не имеет смысла».

После closure spec не модифицируется (frozen invariant). Бэклог обновляется через режим update.

### Переиспользование поддерживающих функций между итерациями (Шаг 5 detail)

Расширенное дерево решений для каждой новой domain-функции на Шаге 5 — см. Шаг 5 § 3 выше.

Указатель на shipped-domain (минимальный) обычно используется только для дедупликации по названию в Шаге 4. При запросе по scope — чтение спеки по требованию (редкий кейс).

### Автоподсказка из parked-бэклога

Скилл сканирует parked-бэклог на семантическое пересечение в трёх местах:
- **Шаг 4** — после описания оператором кандидата в domain-функцию.
- **Шаг 5** — после кандидата enabler, если shipped-проверка не дала совпадения.
- **Шаг 6** — при обсуждении «выносим в parking».

Эвристика совпадения — семантическая (LLM-суждение), не regex. Признаки совпадения: близкие названия, перекрывающиеся границы, одинаковые upstream-зависимости.

Ложные срабатывания / пропуски — оператор корректирует устной фразой («нет, другая фича»), не блокирует процесс.

Аналогично для **deferred-развилок** в CCP3 — совпадение → подсветить «связано с deferred Sf-Y из iterN, резолвить сейчас или продолжать как отдельную развилку?».

### Режим update (D14 mechanics)

Если на Session setup определена активная спека → режим update применяется к текущей спеке и бэклогу.

**Поддерживаемые операции:**

- **Добавить функцию** (новый F-X): probe Шаги 4-6 для новой; Шаг 5 enabler closure; обновление спеки.
- **Удалить функцию**: проверка downstream-зависимостей (никто не зависит?); если зависят — резолвить.
- **Разделить функцию** (X → A + B): добавить A + добавить B; удалить X; перенаправить зависимости; пересмотр целостности.
- **Слить функции** (A + B → C): добавить merged; удалить обе; перенаправить зависимости.
- **Обновить scope cuts**: редактировать карточки (поля Входит / Не входит), без каскада.
- **Добавить / отредактировать CC-политику**: новая или обновлённая запись в Section 6.
- **Резолвить OPEN-развилку**: применить изменения по выбранной ветке (add/remove функций); обновление списка функций.

**Mechanics:**
- Read tool загружает спеку + бэклог в контекст.
- LLM понимает структуру семантически, редактирует точечно через Edit tool.
- Проверка целостности после каждой операции (сирота / разорванная замкнутость / циклы).
- Mermaid в Section 4 регенерируется LLM'ом при изменении зависимостей.
- Представление diff'а — через «было/стало» в предложении оператору перед применением.
- Гейт подтверждения — на логическую операцию (split = одно подтверждение, не три отдельных).

**Без structured parsing.** Скилл работает с markdown как с текстом. Mermaid helper script alternative припаркован в OQ14 (если LLM-генерация Mermaid окажется ненадёжной на больших графах — пересмотрим).

**Frozen invariant** — shipped-спеки не редактируются (D18). Если оператор пытается «открыть замороженную спеку для дополнения» — отказать: «Frozen invariant. Если решение из iter1 нужно пересмотреть — это новое решение в новой спеке, latest wins».

### Поле status в спеке (D18)

Метаданные спеки содержат поле `status` с тремя значениями:

- `draft` — спека в дизайне (оператор работает в текущей сессии).
- `signed` — дизайн закрыт, перешли к feature-spec / impl. Активная спека, можно редактировать.
- `shipped` — итерация отгружена, спека **заморожена**, не редактируется.

**Переходы статуса** (детерминированные действия скилла, без эскалации оператору):

- `draft → signed` — автоматически по завершении Шага 12 + User Review Gate.
- `signed → shipped` — по явной команде оператора «iter X отгружена / iter X в production». Скилл применяет детерминированно.
- `shipped → anything` — запрещено. Скилл блокирует попытки редактирования.
- `signed → draft` — оператор может вручную через правку метаданных, если решает переоткрыть дизайн (скилл не борется, но и не инициирует).

Активная = `draft` или `signed`. Замороженная = `shipped`.

**Связь с дисциплиной P6 ARCHITECTURE.md.** Переходы статуса, проверки целостности, валидация формата и миграция в бэклог — детерминированные действия. Скилл выполняет их без вопросов оператору. Эскалация — только для продуктовых решений (что включаем, как режем scope, как резолвим конфликт).

## Output Format

### Расположение файлов

- Spec: `<project>/docs/specs/YYYY-MM-DD-<product-slug>-<iteration-slug>-product-spec.md`
- Backlog: `<project>/docs/specs/<product-slug>-product-backlog.md`

(Конкретное расположение по умолчанию зависит от OQ15 file convention для семейства hi_flow — на момент imp v0.1.0 по умолчанию — плоский `docs/specs/`.)

### Skeleton

Полный skeleton спеки — `references/product-spec-template.md`. Полный skeleton бэклога — `references/product-backlog-template.md`. Worked example — `references/example-contact-tracker-mvp.md`. Скилл читает эти три файла на старте сессии для анкоринга формата output'а.

### Принцип PRD-as-standalone

Spec самодостаточна для контекста итерации. Бэклог читается отдельно для памяти между итерациями. ARCHITECTURE.md проекта читается отдельно для архитектурного контекста — не дублируется.

### Принцип plain language

Spec и бэклог адресованы продуктологу, не разработчику. По умолчанию — продуктовый русский. Англицизм допустим только если выполняется один из трёх критериев (см. memory `feedback_plain_language_conditional.md`):

1. Устоявшийся термин в индустрии (MVP, KPI, API, UX, ROI), у которого нет распространённого русского аналога.
2. Объективно компактнее или точнее русского при том же смысле, без потери нюанса.
3. Игра слов или специфический жаргон, теряющий смысл при переводе.

Чисто инженерный жаргон (extract-all, payload, throughput, idempotency, fallback) — переводится или раскрывается. Понятные продуктовые имена переменных (target_weight, current_weight) — нормально как структурные идентификаторы.

Контроль — гештальт-просмотр готового текста: читается ли это как русский с редкими оправданными вкраплениями, или как англо-русская мешанина? Опасная зона — кластеризация двух+ англицизмов в одном предложении при наличии русских эквивалентов.

### Адаптивная глубина

Шаги 1-3 (Описание → Группы → Задачи) адаптируют глубину по типу проекта из Шага 1.2:
- Личный продукт: 1 группа, тривиальный список задач.
- Стартап-MVP / open-source: 2-4 гипотетические персоны, гипотетические задачи.
- Внутренний инструмент / коммерческий: 3-7+ реальных ролей, матрица «группа × задачи», **buyer outcomes** в Шаге 1.4.

## Operational Rules — что скилл обеспечивает

1. **Только опрос сверху вниз.** Шаг 5 (вывод поддерживающих функций) идёт **из** пользовательских функций через продуктовый язык probe-таблицы D9, не через инженерный «нужен ли auth?». Скилл переводит инженерный вопрос в продуктовый эквивалент перед задаванием оператору.

2. **D2 трёхтестовый гейт для каждой функции-кандидата.** Independent shippability + atomic user job + negotiability. Провал любого теста → переразложить (split / merge / drop), не пускать в список функций.

3. **D5 boundary test для каждой развилки-кандидата.** Меняет состав функций или связи между ними? → стратегическая развилка (Section 5). Иначе — уровень фичи, не записывается (пойдёт в feature-spec).

4. **D17 асимметричная дисциплина указателей в бэклоге.** Shipped-enabler 5-6 строк (Входит/Не входит), shipped-domain 3 строки, parked полное содержимое (любой Type), deferred полное содержимое, rejected one-liner.

5. **D18 frozen invariant для shipped-спек.** Никаких правок. Переопределение через новую спеку, latest wins.

6. **Дисциплина эскалации к оператору (P6).** Не задавать оператору вопросы по детерминированным действиям (переходы статуса, проверки целостности, валидация формата, миграция в бэклог) — выполнять без вопроса. Не принимать решения за оператора в продуктовых вопросах — эскалировать.

7. **Закрытие требует резолва всех стратегических развилок.** OPEN-развилки недопустимы при closure — должны быть либо RESOLVED, либо DEFERRED (мигрируют в бэклог), либо OUT-OF-SCOPE.

8. **Mermaid обязателен в Section 4.** Регенерация LLM'ом при каждом изменении поля `Зависит от` в карточках. Источник истины — поле `Зависит от` в карточках.

9. **Living document scope.** Spec — living **внутри** одной итерации (промежуточный feedback от feature-spec → режим update). Бэклог — living **через** итерации.

10. **Правило эскалации.** При неоднозначности, неразрешимой из доступных данных — эскалировать оператору, не перезапускать upstream-скилл.

## Format Rules

1. **Иерархические ID (Cockburn-style).** Функции: `F-<module-slug>-N` (например, F-id-1, F-comm-1, F-sales-2). Стратегические развилки: `Sf1, Sf2 ...`. Сквозные политики: `CC1, CC2 ...`. Переиспользуемые подполитики: `P-<UPPER-CASE-WITH-DASHES>`.

2. **ID назначен ⇔ карточка существует.** Нет висящих ID без карточек. Нет карточек без ID.

3. **Поля `Status` и `Тип` обязательны на каждой карточке функции.** Status ∈ {in-scope, parked, out-of-scope}. Тип ∈ {пользовательская, поддерживающая}.

4. **Тег кардинальности (XOR / OR / OPT)** обязателен на стратегических развилках с ветвями.

5. **`Зависит от`** — обязательное поле для **всех** функций (для domain-функций часто пусто или ссылки на enabler'ы; для enabler'ов часто пусто или ссылки на другие enabler'ы). Mermaid строится по нему.

6. **`Inherits`** — опциональное поле в карточке для standing-политик из предыдущих итераций.

7. **`Доступно группам`** — обязательное если Шаг 9 сработал и группы различаются. Иначе опускается.

8. **Все стратегические развилки при closure** — RESOLVED / DEFERRED / OUT-OF-SCOPE. OPEN-развилки при closure — блокирующая ошибка.

9. **Нет резюме в конце спеки** — оператор читает в полном объёме по принципу PRD-as-standalone.

10. **Имя файла бэклога** — `<product-slug>-product-backlog.md` (без даты, один на продукт).

11. **Имя файла спеки** — `YYYY-MM-DD-<product-slug>-<iteration-slug>-product-spec.md`.

## Implementation Notes

- На старте сессии читать reference-файлы (шаблоны + пример) для анкоринга формата output'а.
- При первой итерации продукта (нет существующего бэклога) — создать бэклог по skeleton'у `product-backlog-template.md`. В режиме update — загрузить существующий бэклог в контекст.
- При новой стратегической развилке — применять тест D5 на всё, что выглядит как развилка. Если меняет состав функций / зависимости → стратегическая развилка в Section 5. Иначе — отбросить (уровень фичи, пойдёт в feature-spec, в product-spec не записываем).
- При завершении Шага 12 — детерминированно перевести status `draft` → `signed` (без вопроса оператору).
- Никаких самооценочных предложений на старте (D13). Если оператор просит direct — вежливо отказать и предложить короткий brainstorm-путь.
