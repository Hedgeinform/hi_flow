# Goal Setting — постановка начальной цели по весу

> **Note (meta).** Это пример-спека, иллюстрирующая формат, спроектированный для скилла продуктовой спеки на уровне фичи. Используется как стресс-тест формата на реальном кейсе с заметной продуктовой сложностью. Реальная Zhenka может иметь другие конкретные значения и приоритеты — здесь иллюстрация, не финальный продакшн-договор.

## Sample dialogs

### Happy path: реалистичная цель принята как есть

```
User:  хочу к лету сбросить вес
Bot:   окей, к лету понял. На сколько хочешь похудеть?
       "на X кг" или "до Y кг".
User:  на 5 кг
Bot:   хорошо. Целевой вес 65 кг (с текущих 70), целевая дата 1 июня.
       Темп ~0.7 кг/нед — в рамках безопасной нормы. Записываю.
       Удачи!
```

### Corrected path: нереалистичная цель, корректировка принята

```
User:  хочу к 1 мая весить 60
Bot:   минутку. Сейчас 75 кг, до 1 мая 5 недель.
       Это темп 3 кг/нед — больше биологически безопасной нормы
       (~1 кг/нед). Могу предложить:
       — удлинить срок: 60 кг к началу августа
       — снизить цель: 70 кг к 1 мая
       Что выбираешь?
User:  давай 70 к 1 мая
Bot:   записал: цель 70 кг к 1 мая. Темп ~1 кг/нед.
       Удачи!
```

### Refused path: медицинская опасность

```
User:  хочу весить 45
Bot:   при твоём росте 170 см это даст ИМТ ~15.5 —
       зона критической недостаточности веса.
       Я не могу помочь выстроить такую цель — это потенциально
       опасно для здоровья. Если у тебя есть такие планы,
       обсуди с врачом, он подберёт безопасный план.
       Если хочешь, помогу поставить цель в более здоровом
       диапазоне (55-65 кг).
```

## Цель

Универсальная начальная постановка цели по весу для пользователя Zhenka (фитнес-бота). Помочь оформить цель с учётом реалистичности по биологическим нормам и безопасности по медицинским границам.

**Out of scope:**

- Цели по мышечной массе или композиции тела (фокус только на вес).
- Цели по конкретным калориям / макро / физической активности (отдельный flow).
- Постановка цели для пользователей с медицинскими условиями (расстройства пищевого поведения, эндокринные нарушения) — направление к специалисту, см. CC1, CC2.

---

## Контракт входа

### От пользователя (raw)

- **target_weight** — опционально. Формы:
  - абсолютное число (`50`, `65 кг`, `70`)
  - дельта от текущего (`на 5 кг`, `-3`, `плюс 7`)
  - процент (`на 5%`, `на -10%`)
- **target_deadline** — опционально. Формы:
  - точная дата (`1 мая`, `15.06.2026`)
  - сезонная ссылка (`к лету`, `к зиме`, `к новому году`)
  - длительность (`за 3 месяца`, `через полгода`)

### Из контекста

- **current_weight** — число (кг), из профиля, обязательно. При отсутствии — фича не активируется, бот сначала просит заполнить профиль.
- **height** — число (см), из профиля, обязательно (для расчёта ИМТ).
- **age** — число, из профиля, обязательно (для расчёта безопасного темпа изменения веса).
- **sex** — male/female, из профиля, обязательно (для того же расчёта).
- **medical_flags** — массив маркеров из профиля (`eating_disorder`, `endocrine_disorder`, ...). Может быть пустым. См. CC2.

---

## Контракт выхода

- **Запись в БД:**
  - `goal` (id, user_id, target_weight_kg, target_deadline_date, original_weight_input_text, original_deadline_input_text, motivation_pattern, created_at, status, smoothing_applied, smoothing_reason_text). Status: `active | achieved | abandoned | expired | replaced`.
  - `goal_history` (история изменений) — только дополнение, не перезаписывается при замене или истечении.
- **Возврат пользователю:** подтверждение с финальными нормализованными значениями + расчётный темп + поддерживающий тон.
- **Side effects:**
  - При принятии цели: расписание напоминаний (см. P-REMINDER-SCHEDULE).
  - При отказе по CC1/CC2: лог события для последующего мониторинга (анонимизированный).

## Развилки

### F0. Стратегия чтения первого сообщения [decision: распознать всё сразу или собирать по полям] [status: RESOLVED]

**Resolution:** распознавать всё сразу — бот разбирает первое сообщение пользователя, извлекает все упомянутые поля (target_weight, target_deadline, намёки на мотивацию), затем дополнительно спрашивает только то, чего не хватает.

**Branches [XOR]:**
- F0.a — распознать всё сразу (выбран): пользователь не повторяет себя, диалог ощущается естественно. END (далее → F1).
- F0.b — собирать по одному полю, игнорируя первое сообщение: каждое поле спрашивается отдельно, первая фраза используется только как сигнал начать. **Отклонён:** игнорирует пользовательский ввод, бот выглядит тупо.
- F0.c — гибрид: распознавать только точные числа и даты, не разбирать формы вроде «к лету» или «на 5%». **Отклонён:** половинчатое решение, не даёт полного выигрыша по UX.

**Reasoning:** базовый принцип общения — бот не переспрашивает уже сказанное. Игнорирование изначального ввода создаёт ощущение тупости и подрывает доверие.

**Examples:**
- «хочу к лету сбросить 5 кг» → распознан срок «к лету» + изменение веса -5 кг. Подтверждает без дополнительных вопросов. END.
- «хочу к лету сбросить вес» → распознан срок «к лету». Спрашивает вес. → F1.2.
- «хочу 5 кг сбросить» → распознано изменение веса -5 кг. Спрашивает срок. → F1.1.
- «хочу похудеть» → ничего конкретного, спрашивает оба поля. → F1 (`-,-` ветка).

**Связи:** F1 (проверка полноты после распознавания), F2 (нормализация применяется к распознанным полям перед F1.3).

**Открыто:** устойчивость распознавания на нестандартном вводе. Это в основном вопрос реализации (Phase 2/3); при сбое распознавания → откат на собирание по одному. Не блокирует Phase 2.

---

### F1. Полнота входа от пользователя [decision: оба обязательны или оба опциональны] [status: RESOLVED]

**Resolution:** оба поля опциональны, но требуется минимум одно. Пользователь не всегда знает оба сразу.

**Branches [XOR]** (decision table):

| has_weight | has_deadline | branch | действие                                                |
|------------|--------------|--------|--------------------------------------------------------|
| -          | -            | —      | спросить хотя бы одно поле. END.                        |
| +          | -            | F1.1   | записать «до X кг, без срока»                           |
| -          | +            | F1.2   | записать «к сроку, без целевого веса»                   |
| +          | +            | F1.3   | проверить реалистичность → see F1.3                     |

**Examples:** «60 кг неважно когда» (F1.1); «к лету неважно сколько» (F1.2); «10 кг за 2 мес» (F1.3 → F1.3.2).

---

### F1.1. Только вес дан [status: RESOLVED]

**Resolution:** записать как бессрочную цель. Темп не считается. Напоминания о прогрессе раз в 2 недели (P-REMINDER-SCHEDULE, режим «без срока»). END.

**Examples:** «до 60 кг» → goal.target_weight=60, goal.target_deadline=null, goal.motivation_pattern="open-ended".

---

### F1.2. Только срок дан [status: RESOLVED]

**Resolution:** записать цель со сроком, но без числового веса. Прогресс измеряется как «движение в направлении» (теряет / набирает от текущего). Бот не корректирует, отслеживает изменения. END.

**Examples:** «к лету» → goal.target_deadline=2026-06-01, goal.target_weight=null, goal.motivation_pattern="movement".

---

### F1.3. Дан и вес и срок [decision: реалистичность темпа] [status: RESOLVED]

**Resolution:** проверка безопасного темпа изменения веса по биологическим нормам (с учётом age, sex, текущий ИМТ). Граница ≈ 1% веса в неделю, корректируется по таблице ВОЗ (см. P-REALISM-CHECK).

**Branches [XOR]:**
- F1.3.1 — реалистично (темп не превышает границу) → принять цель как есть. END.
- F1.3.2 — нереалистично (темп превышает границу) → предложить корректировку → see F1.3.2.

**Связи:** CC1 применяется до F1.3.

---

### F1.3.1. Реалистичная цель [status: RESOLVED]

**Resolution:** записать значения как есть, подтвердить пользователю с расчётным темпом для прозрачности. Назначить напоминания по P-REMINDER-SCHEDULE (режим «со сроком»). END.

---

### F1.3.2. Нереалистичная цель — корректировка [decision: какой параметр смягчить] [status: RESOLVED]

**Resolution:** зависит от отношения запрошенного срока к минимально возможному безопасному:

- Если запрошенный срок < 50% от безопасного → смягчаем **вес** (срок остаётся как есть).
- Если запрошенный срок ≥ 50% → предлагаем **оба варианта** на выбор: либо удлинить срок до безопасного, либо снизить целевой вес до достижимого за указанный срок.

**Reasoning:** очень короткий срок делает смягчение срока тривиальным (даёт 1-1.5 кг за месяц вместо 10) — обесценивает результат. Умеренное превышение даёт пользователю содержательный выбор.

**Branches [XOR]:**
- F1.3.2.1 — пользователь принимает корректировку → принять скорректированную цель. END.
- F1.3.2.2 — пользователь отказывается → see P-INSIST-HANDLING.
- F1.3.2.3 — пользователь предлагает свою корректировку, отличную от предложенных вариантов → запустить F1.3 заново с новыми параметрами.

**Открыто:** формальная граница «50%» — нужно сверить с нутрициологом или валидировать на 5-10 реальных кейсах.

**Examples:**
- «10 кг за 2 мес», безопасный темп ~2.5 кг/мес → безопасный срок ~4 мес. Запрошенный 2 < 4×0.5=2, граничный → смягчаем вес (~1.5 кг за 2 мес).
- «15 кг за 4 мес», безопасный срок ~12 мес. Запрошенный 4 < 12×0.5=6 → смягчаем вес. *(Но в реальном обсуждении кейс был решён «оба варианта»; нужна реальная калибровка.)*

---

### F1.3.2.1. Корректировка принята [status: RESOLVED]

**Resolution:** записать скорректированную цель. `smoothing_applied=true`, `smoothing_reason_text` фиксирует логику. Подтверждение пользователю с расчётным темпом. END.

---

### F1.3.2.2. Корректировка отвергнута → P-INSIST-HANDLING [decision: отказ или принятие с напоминаниями] [status: OPEN]

**Resolution:** OPEN — нужно решение.

**Options:**
- (a) **Полный отказ** — «не возьму на себя такую ответственность; либо в рамках безопасных границ, либо никак».
- (b) **Принять с напоминаниями** — записать запрошенные пользователем параметры, но проактивно отслеживать и напоминать (см. P-REMINDER-SCHEDULE, режим «агрессивный темп»).

**Trade-offs:**
- (a) — последовательно по принципу «бот не врёт о невозможном», но потенциально теряет пользователя категоричного типа. Если пользователь категоричен, лучше иметь хоть какой-то контакт с ним, чем полный отказ.
- (b) — даёт пользователю выбор и наблюдение, но создаёт ответственность бота за поддержание неустранимо проблематичной цели. Морально серая зона.

**Связи:** P-INSIST-HANDLING (детали обработки), P-REMINDER-SCHEDULE (если выбран (b)).

**Открыто:** также — критерий, по которому в (b) бот понимает «пора форсировать пересмотр» (например, отставание от темпа на >X% за Y недель → запуск процесса пересмотра цели).

---

### F1.3.2.3. Пользователь предлагает свою корректировку [status: RESOLVED]

**Resolution:** запустить F1.3 заново с новыми (target_weight, target_deadline). Перепроверить реалистичность. Если опять нереалистично — снова F1.3.2 (но не зацикливаемся бесконечно — после 3 итераций предлагаем P-INSIST-HANDLING). END (после повторного входа).

**Examples:** «не нравятся ни 1.5 кг за 2 мес, ни 10 кг за 6 мес. Давай 7 кг за 4 мес» → перепроверка.

---

### F2. Нормализация неточных форм пользовательского ввода [status: RESOLVED]

Нормализация форм происходит до F1.3. См. P-NORMALIZATION-DEADLINE и P-NORMALIZATION-WEIGHT.

**Sub-forks:**
- F2.1 → see P-NORMALIZATION-DEADLINE.
- F2.2 → see P-NORMALIZATION-WEIGHT.

---

### F-life-1. Цель достигнута [status: RESOLVED]

**Когда:** current_weight достиг target_weight (с допуском ±0.5 кг) до target_deadline.
**Resolution:** празднование + предложение поставить цель поддержания или новую цель похудения / набора. goal.status=achieved. END.

---

### F-life-2. Срок цели прошёл, цель не достигнута [status: RESOLVED]

**Когда:** текущая дата > target_deadline AND |current_weight - target_weight| > 0.5 кг.
**Resolution:** спросить пользователя — продлить срок / переформулировать / отказаться. По умолчанию через 3 дня без ответа → goal.status=expired автоматически. END.

---

### F-life-3. Пользователь меняет цель в процессе [status: RESOLVED]

**Когда:** пользователь явно говорит «хочу поменять цель» / «сменим цель».
**Resolution:** старая goal.status=replaced, история изменений сохранена. Запустить flow Phase 1 заново с current_weight как новой точкой отсчёта. END (после повторного входа).

---

### F-life-4. Пользователь молча забрасывает цель [status: DEFERRED]

**Когда:** пользователь не логирует прогресс ≥14 дней.
**Resolution:** DEFERRED — поведение зависит от дизайна P-REMINDER-SCHEDULE, который сам OPEN. После закрытия P-REMINDER-SCHEDULE этот fork автоматически закрывается.

### F-disamb-1. «Настойчивость» (F1.3.2.2) vs «медицинская опасность» (CC1) [decision: критерий разграничения] [status: OPEN]

**Когда:** пользователь продолжает настаивать на цели, которая выглядит экстремальной.

**Внешне похожие ситуации:**
- F1.3.2.2: пользователь хочет агрессивный темп (5 кг/мес вместо 1 кг/мес), но target_weight в безопасном диапазоне.
- CC1: пользователь хочет target_weight, дающий ИМТ < 16 — медицински опасное независимо от темпа.

**Resolution:** OPEN.

**Кандидатные критерии разграничения** (требуют формализации):
- Расчёт ИМТ: target_BMI = target_weight / (height_m^2). Если < 16 → CC1, иначе → F1.3.2.2.
- Маркеры из профиля: при наличии `eating_disorder` или `endocrine_disorder` → форсируем CC2 раньше всех остальных проверок.
- Возрастной критерий: для подростков (<18) более строгие границы — но age <18 пока out of scope (см. CC3).

**Открыто:** финальный критерий + где он применяется (на каком уровне делается проверка — до F1.3 или после).

**Связи:** CC1, CC2.

---

### F-integ-1. Влияние цели на handle_meal_input [status: RESOLVED]

**Когда:** пользователь логирует приём пищи через инструмент handle_meal_input.
**Resolution:** при наличии активной цели handle_meal_input включает в ответ краткий статус прогресса (current_weight vs target, дней осталось). При отсутствии цели — обычное подтверждение приёма пищи. END.

**Связи:** handle_meal_input в Module Map.

---

### F-integ-2. Влияние цели на fetch_day_summary [status: RESOLVED]

**Resolution:** day_summary всегда включает блок статуса цели (если активная цель есть): прогресс, темп vs план, оставшийся срок. END.

---

### F-integ-3. reply_casually и упоминание цели [status: DEFERRED]

**Resolution:** DEFERRED — не блокер. Решим по итогам наблюдения: упоминать ли цель в случайных репликах или только при явном запросе. Phase 2 может архитектурно поддержать оба варианта без дополнительных решений сейчас.

## Cross-cutting policies

### CC1. Медицински опасный целевой вес [orthogonal] [status: RESOLVED]

**Pre-empts:** F1.3.*, F1.1, F1.2 (любая попытка зафиксировать цель).

**Resolution:** жёсткий отказ + направление к врачу + предложение постановки цели в безопасном диапазоне.

**Pattern:** *Если target_BMI < 16, ИЛИ запрошенный вес требует темпа > 2× безопасного при текущем ИМТ < 18 → отказать в записи цели, залогировать событие, рекомендовать консультацию врача, предложить цель в безопасном диапазоне.*

**Связи:** F-disamb-1 (критерий различения с F1.3.2.2).

---

### CC2. Медицинские маркеры в профиле [orthogonal] [status: RESOLVED]

**Pre-empts:** всё goal setting независимо от значений.

**Resolution:** при наличии `eating_disorder` или `endocrine_disorder` в profile.medical_flags — направление к специалисту, без активации flow целеполагания.

**Pattern:** *Если profile.medical_flags содержит «eating_disorder» ИЛИ «endocrine_disorder» → не запускать flow целеполагания, рекомендовать консультацию специалиста, предложить общую поддержку вместо постановки цели.*

---

### CC3. Возраст пользователя <18 [orthogonal] [status: OUT-OF-SCOPE]

**Note:** Zhenka регистрация требует возраст ≥18 на signup. Этот fork формально out-of-scope для текущей версии. Если в будущем добавится поддержка подростков — должна быть переоценена с другими нормами безопасного темпа.

## Reusable sub-policies

### P-NORMALIZATION-DEADLINE [status: RESOLVED]

Преобразование неточного срока в дату.

| форма ввода        | алгоритм                                                          | пример                       |
|--------------------|-------------------------------------------------------------------|------------------------------|
| точная дата        | разобрать как есть                                                | «1 мая» → 2026-05-01         |
| сезон              | начало сезона ближайшего будущего года                            | «к лету» в апреле → 2026-06-01 |
| длительность       | текущая дата + длительность                                       | «за 3 месяца» → сегодня + 90 дней |

**Reasoning:** «к» в русском обычно означает «успеть к», не «уложиться в». Поэтому начало сезона.

**Used in:** F2.1, и автоматически при любом вводе пользователя.

### P-NORMALIZATION-WEIGHT [status: RESOLVED]

Преобразование относительного веса в абсолютный target_weight.

| форма ввода        | алгоритм                          | пример                       |
|--------------------|------------------------------------|------------------------------|
| абсолютное число   | как есть                           | «65 кг» → target=65          |
| дельта             | current_weight + дельта            | «на 5 кг» с current=70 → 65  |
| процент            | current_weight × (1 + дельта)      | «на -10%» с current=70 → 63  |

**Direction inference:** в контексте Zhenka (фитнес-бот, преимущественно используется для похудения) дельта без явного знака считается отрицательной. При неоднозначности — спрашивать у пользователя.

**Used in:** F2.2, и автоматически.

### P-REALISM-CHECK [status: RESOLVED]

Расчёт безопасного темпа изменения веса и проверка реалистичности.

**Алгоритм:**
- safe_rate_per_week = 0.01 × current_weight × adjustment_factor
- adjustment_factor зависит от текущего ИМТ:
  - ИМТ < 18.5 — 0.5 (более консервативно — пользователь уже худой)
  - 18.5 ≤ ИМТ < 25 — 1.0 (базовая норма)
  - 25 ≤ ИМТ < 30 — 1.0
  - ИМТ ≥ 30 — 1.2 (немного более агрессивно допустимо)
- safe_total_weeks = |target_weight - current_weight| / safe_rate_per_week
- requested_weeks = (target_deadline - сегодня) в неделях
- реалистично если requested_weeks ≥ safe_total_weeks

**Used in:** F1.3, F-life-2.

**Открыто (уточнение):** модификаторы по age и sex в adjustment_factor. Возможно нужны вариации, но базовой формулы достаточно для первой версии.

- Содержание напоминаний — мотивирующее vs нейтральное vs техническое.

### P-INSIST-HANDLING [status: OPEN — наследует F1.3.2.2]

Обработка случая, когда пользователь отвергает корректировку и настаивает на своём.

**Resolution:** OPEN — зависит от выбора в F1.3.2.2 (отказ или принятие с напоминаниями).

**Если выбран отказ:**
- Тон: твёрдый, не агрессивный. Объяснение почему именно эта граница.
- Альтернативное предложение: безопасная цель с расчётом результата за тот же срок.
- Возврат к flow: пользователь может вернуться позже, политика та же.

**Если выбрано принятие с напоминаниями:**
- Принимаем запрошенные параметры.
- Назначаем напоминания по P-REMINDER-SCHEDULE, режим «агрессивный темп».
- Тон: поддерживающий, но честный («твоя цель амбициозна, я буду рядом и подсвечу если будет видно отставание»).

**Used in:** F1.3.2.2.

## Premortem findings (closing probe absorbed)

«Через 3 месяца после запуска — какие жалобы пользователей?»

1. **«Бот заставил меня поменять цель, я этого не хотел»** — отражено в F1.3.2.2 (OPEN).
2. **«Я пропустил срок, бот удалил мою цель без предупреждения»** — закрыто в F-life-2 (3 дня без ответа → expired).
3. **«Бот не учёл, что я беременна»** — пробел в данных профиля. Не закрыто текущей спекой; нужен новый медицинский маркер, но это отдельная фича и отдельная спека.
4. **«Я хотел набрать массу, бот предположил, что я худею»** — частично закрыто в P-NORMALIZATION-WEIGHT (по умолчанию направление = похудение), но при неоднозначности нужно спрашивать. Поднято в P-NORMALIZATION-WEIGHT.
5. **«Бот нудит напоминаниями»** — DEFERRED в P-REMINDER-SCHEDULE (нужен баланс с предпочтениями пользователя).
